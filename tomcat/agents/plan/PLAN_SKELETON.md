## 1. 背景

**说人话**：先把“为什么做、差在哪、边界到哪”讲清楚，再往后写方案和决策。

### 1.1 当前基线 / 现状与差距

**说人话**：这张表用来交代现在是什么样、想改成什么样，别让读者自己脑补差距。

| 项 | 当前情况 | 目标 / 差距 | 说人话 |
| :--- | :--- | :--- | :--- |
| 现状 | <当前行为> | <目标行为> | <为什么要改> |
| 约束 | <协议 / 性能 / 兼容限制> | <本期处理方式> | <哪些事现在不能乱动> |

### 1.2 单一事实来源 / 阶段边界

**说人话**：这里是防止范围越写越大，也防止实现时和真正的权威文档打架。

| 项 | 内容 | 说人话 |
| :--- | :--- | :--- |
| 权威文档 | `<docs/...>` / `<tasks/...>` | <冲突时以谁为准> |
| 本期范围 | <本期做什么> | <这次真正交付什么> |
| 非目标 | <本期不做什么> | <哪些事不要误以为一起做了> |

### 1.3 Bug 补充（若适用）

**说人话**：如果这是修 bug，就别只写“有问题”，要把现象、期望和差多少讲实。

| 要素 | 内容 | 说人话 |
| :--- | :--- | :--- |
| 现象 | <用户可见异常 + 数据> | <不是“感觉不对”，而是哪里真的不对> |
| 期望 | <正确行为> | <修完后应该长什么样> |
| 量化偏差 | <异常值 vs 期望值> | <差多少决定怎么查> |
| 对比组 | <正常对照 / 重启后 / 另一条路径> | <证明这是机制问题不是偶发> |

#### Step-by-Step（若适用）

**说人话**：如果 bug 跨多个时间步骤才显现，就按时间线把它怎么一步步坏掉写出来。

```text

- 关键变量: ...
- 发生了什么: ...

Step 2: ...
```

**背景小结（说人话）**：<一句话收口：这件事的背景、边界、这期的真正目标。>

## 5. 测试方案与验收

**说人话**：这一章回答的是“怎么证明做完了”，不是“作者主观觉得差不多了”。

**说人话**：先把不同层级测试各自负责什么分清，别把所有断言都堆到一个地方。

| 类型 | 场景 | 预期 | 落点 | 说人话 |
| :--- | :--- | :--- | :--- | :--- |
| 单元测试 | <正常路径 / 边界> | <断言> | `<tests/...>` | <这一层主要卡什么问题> |
| 集成测试 | <模块联动> | <断言> | `<tests/...>` | <验证接口是否真的串起来> |
