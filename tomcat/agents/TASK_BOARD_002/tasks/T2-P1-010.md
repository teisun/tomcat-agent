- [`payload.rs`](../../../src/core/llm/openai_responses/payload.rs) 只 replay assistant `output_text` + `function_call`，**不带** reasoning item（无 `encrypted_content` / `thinking_signature`）

#### hermes 胶水层三件套（路线 D 方案说明）

hermes 不自己做 thinking UI，而在 **API ↔ 内部消息 ↔ 下游** 之间做三层粘合（[thinking 显示整改 plan](/Users/yankeben/.cursor/plans/thinking显示整改_424c3911.plan.md) §4）：

| # | 环节 | 做什么 | 说人话 | Tomcat 若采纳 D 的落点（示意） |
|---|------|--------|--------|-------------------------------|
| **1** | **reasoning item → 内部消息格式** | Responses `output[]` 里 `type=reasoning` 的 item 规范化进 assistant 内部结构（与 `output_text` / `function_call` 分列）；落盘字段如 `codex_reasoning_items`、可选 `reasoning` 摘要文本 | 把 API 形状翻译成 Tomcat 能存、能 replay 的消息形态 | `openai_responses/payload.rs` 响应 normalize；`ChatMessage` / transcript schema 扩展 |
| **2** | **`encrypted_content` replay** | 捕获 `encrypted_content` + `summary[]`，下轮组 `input[]` 时 **原样 push** reasoning item（`store: false` 亦可）；遵守 reasoning 后须有 following item | 把「思维链钥匙」还给 API | 同路线 B；[`codex_responses_adapter.py`](../../../../hermes-agent/agent/codex_responses_adapter.py) `_chat_messages_to_responses_input` |
| **3** | **`<think>` scrubber** | 对流式 assistant 正文做 **有状态** 标签过滤（`<thinking>` / `<think>` 等），防止非 Responses 模型把 raw CoT 漏进可见正文 | 有些模型把 thinking 塞进正文标签，要在 upstream 抠掉 | 新模块或 [`think_scrubber.py`](../../../../hermes-agent/agent/think_scrubber.py) 等价 Rust 状态机；挂在 **正文 delta 路径**（非 `[thinking]` summary 通道） |

```text
OpenAI Responses / chat.completions
        |
        v
  [1] normalize reasoning item -> 内部 assistant 消息（codex_reasoning_items 等）
        |
        +--> [2] 下轮 replay encrypted_content -> input[]
        |
        +--> assistant 正文流
                |
                v
          [3] StreamingThinkScrubber -> 下游 CLI/落盘只见干净正文
```

**路线 D 与 Tomcat 关系**：Tomcat 是 **一体式 CLI agent**，不必抄 hermes「渲染交给下游」；可 **只摘 1+2+3 中需要的段**（例如已有 Responses summary 通道时，**3 主要针对 chat.completions / 网关泄漏**）。认领时在 design 写明 D 是全做还是 **D-lite**（仅 1+2，scrubber 另开任务）。

**路线 D 落地要点（认领时展开）**：

- **normalize（①）**：`output_item.done (reasoning)` / 非流式响应解析时写入内部 reasoning 结构，与可见 `thinking` 文本分离
- **replay（②）**：同路线 B；assistant turn 组装 `input[]` 时先 push reasoning items 再 push message/tool
- **scrubber（③）**：`content_delta` 或等价正文流入口包一层状态机；跨 delta 缓冲未闭合标签；turn 结束 `flush`；**不**替代 `stream.rs` summary 去重或 CLI 折叠标题

**路线 C 落地要点（认领时展开）**：

- **capture**：`output_item.done (reasoning)` 或 stream 收口时，`thinking_signature = JSON.stringify(reasoning_item)` 挂到对应 `ThinkingContent` / transcript 条目
- **persist**：随 session jsonl 落盘（可读 `thinking` 文本 + 不透明 signature 并存，UI 折叠不影响 replay）
- **replay**：`build_responses_input` 遍历 assistant history，有 signature 则在同 turn 的 `output_text` / `function_call` 前 push 还原的 reasoning item（遵守 API：reasoning 后须有 following item）
- **与显示解耦**：thinking 显示整改（折叠标题、stream 去重）**不依赖** 本卡；本卡也不改 CLI 折叠逻辑
