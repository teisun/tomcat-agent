### T2-P0-012 | llm-binary-attachments | 图片/二进制文件传给 LLM

| 字段 | 内容 |
|------|------|
| **优先级** | P0 |
| **状态** | `DONE`（**2026-05-12**：CLI / `read` / `read_file` / OpenAI Responses 多模态与类型分流已合入；**TUI 拖拽 chip 子项推迟**，见下文「推迟」） |
| **负责人** | — |
| **分支** | `feature/llm-binary-attachments` |
| **阻塞点** | — |

**目标**：用户拖拽或显式指定图片 / 二进制文件时，系统应把文件作为 LLM attachment 传入模型，而不是走 `read_file` 的 UTF-8 文本路径误读、乱码或污染上下文。

**子项**（具体方案由计划阶段确定，先列待研究方向）：
- [x] 新增文件类型判定层：按 MIME / magic bytes / UTF-8 可解码性区分 text、image、generic binary；文本文件继续走现有 `read_file`，二进制文件禁止静默 UTF-8 解码 — **已落地于 `read` / `read_file` 路径**：① **`detect_inline_mime`**（扩展名 + 头 12 字节 magic）→ `ReadResult::Image` / `Pdf`；② **`read_window_blocking`** 首块含 `\x00` → `Binary` 早返并结构化报错（非 UTF-8 静默读）；③ 窗口字节 **`String::from_utf8`**、**`read_file_impl` → `read_file_utf8`** 遇非法 UTF-8 **显式 `Err`**，无 lossy 解码。见 [`read.rs`](../../../src/core/tools/primitive/executor/read.rs)（`detect_inline_mime` / `read_window_blocking` / `read_impl` / `read_file_impl`）与 [`read.md`](../../../docs/architecture/tools/read.md) **§4.2.4**（PR-RJ / T3 多模态边界）。*注：非图/PDF 的「通用二进制」当前无成功文本通道，仅拒绝 + 提示；若需 `file(1)` 式 sniff 可另开增强项。*
- [x] 设计 `ChatRequest` / message content 的 attachment 抽象：`ChatMessageContentPart` 升级为 `#[serde(tag="type")]` 三态枚举（`InputText` / `InputImage` / `InputFile`），双通道（inline base64 + 已知 file_id）helper 落地于 [`src/core/llm/types.rs`](../../src/core/llm/types.rs)
- [x] Provider 适配（Responses 部分）：[`OpenAiResponsesProvider::part_to_responses_value`](../../src/core/llm/openai_responses.rs) 翻译 `input_image` / `input_file`；Completions（`OpenAiProvider`）入口结构化拒绝并指向 `provider=openai-responses`；Anthropic / Qwen 等其它 provider 的多模态映射等对应 Provider 接入再补
- [x] 安全与预算（inline 部分）：`IMAGE_MAX_BYTES = 4.5 MB` / `FILE_MAX_BYTES = 25 MB` 硬限 + image MIME 白名单 + base64 合法性校验，落地在 helper 入口；transcript metadata-only 与 `PermissionGate` 在 **`read` / provider 路径**已具备；**TUI** 侧复用待 **T2-P0-008**（推迟）。
- [x] 回归测试（wire 部分）：5 个 unit test 覆盖 wire（image_data_url / file_data_url / image_file_id / file_file_id / system 降级）、1 个拒绝测试（Completions）、5 个 helper 失败用例、3 个真 API roundtrip 集成测试（image / pdf / oversize helper）；详见 [`tests/openai_responses_integration_tests.rs`](../../tests/openai_responses_integration_tests.rs) 与 [`tests/fixtures/llm_multimodal/`](../../tests/fixtures/llm_multimodal/)
- [x] **OpenAI Files Upload Manager**（multipart `POST /v1/files` + 生命周期 + reuse cache + 异步 helper）：已从本卡拆出，由 **[T2-P0-015 | llm-files-upload-manager](./T2-P0-015.md)** 跟踪与验收；实现见 [`src/core/llm/openai_files.rs`](../../../src/core/llm/openai_files.rs)、[`ChatMessageContentPart::{image_upload,file_upload}`](../../../src/core/llm/types.rs)、[`tests/openai_files_integration_tests.rs`](../../../tests/openai_files_integration_tests.rs)。

**推迟（不纳入本卡验收；当前无 TUI 宿主）**

- **TUI / 拖拽路径接入**：附件 chip、拖入摘要、与下一轮 `user` 消息并发的交互 — **本阶段不做**。后续由产品具备 TUI 后，在 **[T2-P0-008](./T2-P0-008.md)**（TUI 体验强化）或新开任务中承接；与 **T2-P0-012** 已交付的 `read` + `tool_exec` 注入链正交。

**进度**：**2026-05-05** wire 期闭环；**2026-05-12** 本卡按「无 TUI」范围结项：`read` / `read_file` 分流、Responses 多模态、Files 子系统（**T2-P0-015**）均已落地；上列「推迟」单开排期。

**依赖**：T2-P0-005（工具系统整改后再调整 `read_file`/attachment 边界）；T2-P0-006（LLM content 抽象与 provider 适配先就位）；T2-P0-008（TUI 附件展示与拖拽交互承载）

**被依赖**：—

**协作接口**：
- 消费：`dragged_path`、`PrimitiveExecutor::read_file`、`ChatRequest`、`LlmProvider`、`PermissionGate`
- 提供：`AttachmentContent` / `BinaryAttachment`（命名由计划阶段定）、provider capability 查询、二进制安全读取路径

**验收标准**：
- PNG/JPEG 图片拖入后作为 LLM image attachment 发送，mock provider 可断言 MIME、文件名、bytes/hash
- 任意非 UTF-8 二进制文件不会被 `read_file` 当文本读入上下文；错误信息明确提示可用的附件路径或 provider 限制
- transcript / audit 只记录附件 metadata、hash、大小与路径权限来源，不落原始 bytes
- 多模态不支持场景有单测覆盖，行为为结构化失败而非 silent fallback

> **结项说明**：首条「拖入」验收依赖 **TUI** 宿主；TUI 子项已推迟，当前以 **`read` 工具 + 集成测 / mock provider** 覆盖等价模型输入能力。

**说明（来源溯源）**：本任务承接用户反馈——当前用户想把图片 / 二进制文件交给 LLM 时，容易被系统当作 UTF-8 文本文件读取，造成乱码、token 浪费和模型误判。修复重点是「输入语义分流 + LLM 多模态附件通道」，不是扩大 `read_file` 的文本读取能力。

**验收口径（结项）**：以 **CLI + `read` 工具 + OpenAI Responses** 路径为准；**TUI 专属**交互不在本卡 closure 内（见「推迟」）。
