> 上位规范：[`Constitution.md`](../../Constitution.md)（**二.10「说人话」辅助**）· [`DOCUMENTATION_GUIDE.md`](DOCUMENTATION_GUIDE.md)。

## 1. 何时写一份新方案

任一即触发：

1. 设计跨 ≥ 2 个 `src/` 一级子目录的行为；
2. 新增 / 修改 wire 协议 / 事件 / 错误契约；
3. 引入新的状态机 / 生命周期 / 取消传播；
4. 计划文档（`agents/plan/*`）显式写"开发前需新增架构文档"（参见 [`PLAN_SPEC.md §2`](../../../../agents/plan/PLAN_SPEC.md)）。

模块 README 描述"本模块做什么"；架构方案描述"一件事跨多个模块怎么串起来做"。两者并存、不互相替代。

## 2. 章节骨架（推荐顺序）

| 章节 | 硬度 | 何时必出 |
|------|------|----------|
| §1 术语统一 | **MUST** | 任何易混淆的命名（`interrupt / cancel / abort` 之类） |
| §2 竞品 / 选型对比（**调研**） | **MUST** | 关切、横向对比、维度词典等；**不含**已定稿七列表（已定稿见 **§4**） |
| §3 目标与设计原则 | **MUST** | 所有方案；观察指标、非目标 |
| §4 落地选型与实施（**已定稿**；**至少**含 **§4.1** 落地选型决策表 + **§4.2** 实施点（已闭环）；**§4.0** 章节编排见下） | **MUST** | 所有方案 |
| §5 协议（入参 / 出参 / Schema） | **MUST**（涉及对外接口时） | 引入或修改工具 / 事件 / Trait / wire 时 |
| §6 文件职责总览（One-Glance Map） | **MUST** | 所有方案 |
| §7 调度时序（运行时图） | **SHOULD** | 跨模块数据流 / 多线程 / 异步 / 取消传播 |
| §8 状态机 | **SHOULD** | 有显式生命周期 / 状态迁移 |
| §9 配置与环境变量 | **SHOULD** | 引入新配置项 / env 开关 |
| §10 错误模型 / 截断 / 警告 | **SHOULD** | 有非 panic 的归一化结局（warning / truncated 等） |
| §11 测试矩阵（验收） | **MUST** | 所有方案 |
| §12 风险与应对 | **MUST** | 所有方案 |
| §13 历史决策 / 跨文档修订 | **SHOULD** | 有被取代的旧方案 / 影响相邻方案文档 |

> **行序**：默认 **§1→§13** 即上表顺序——**术语 → 竞品调研 → 目标 → 已定稿选型与实施 → 协议 → One-Glance → …**。标杆实现：[`read.md`](../../../../docs/architecture/tools/read.md)（文档内 `## 1`–`## 5` 与上表语义一一对应）。若某稿因历史原因局部编号不同，**须**在文首说明与上表 § 的对应关系。
>
> 章节编号可酌情合并（如 §7 + §8 合成「调度与状态机」），但 **MUST** 项一个不能省。

## 3. 各节硬约束

### §1 术语统一（MUST）

每条术语三件套：

- **语义**：一句大白话。
- **数据载体**：`#[serde(skip)]` 字段 / `Arc<Mutex<...>>` / 某个 wire 常量等。
- **行为约束**：触发 / 失效 / 唯一性 / 与相邻术语的互斥关系。

若术语以**表格**呈现：在保留三件套的前提下，**SHOULD** 在表末追加 **`说人话`** 列（更短口语，见 **§14.1**）；与「语义」列重复度过高时可合并为一列，但须保留一句纯口语。

模糊时间词（"LLM 回复后"、"发起下一次请求前"）必须在首次出现处用一句话钉死指代位置。

### §2 竞品 / 选型对比（调研）（MUST）

把「读过竞品 / 备选路线后的**调研材料**」钉在 spec 里，避免只在 `agents/plan/*` 口头对齐、正文只剩首屏散文。**关切框图、竞品横向表、维度词典**等仍按主题 **SHOULD** 展开（无成熟对标时可缩短为一段「无可比公开协议」声明 + 自研约束）。

本节**不含**已定稿的 **§4.1** 七列「维度上的取舍」矩阵与 **§4.2** 实施点总表——二者须在 **§4** 给出；此处承上启下，为 §3 目标与 §4 已定稿结论提供证据链。

在有多条候选路线时，仍须给出「为什么选 A 不选 B」的 **3–5 条**理由（可写在 §2 表下或紧随 §4 决策表）。

竞品横向对比模板（§2 正文 **SHOULD** 保留或等价浓缩）：

```markdown
| 竞品 | 形态 | 关键设计 | 我们借鉴的点 | 说人话 |
|------|------|----------|---------------|--------|
| ripgrep | spawn | 默认遵守 .gitignore | Tier1 内容主力 | 默认就尊重 gitignore，跟 rg 习惯一致。 |
| ignore crate | 进程内 | WalkBuilder + filter_entry | Tier2 兜底主力 | 进程内遍历时用成熟库做过滤。 |
```

### §3 目标与设计原则（MUST）

- 用**观察指标表**替代散文："2 秒内子进程死亡"、"truncated=true 时必给 next_offset"。
- 显式列**非目标**及"推给谁 / 哪个任务号"，防止范围蔓延。

```markdown
| 目标 | 观察指标（落地后用户可感知） | 说人话 |
|------|------------------------------|--------|
| G1   | Ctrl+C 后 2 s 内 child 进程死亡 | 用户按停之后，子进程别拖着不死。 |

| 非目标 | 推给 | 说人话 |
|--------|------|--------|
| 跨 session 续跑 | T2-P1-001 | 这条本期不做，交给看板那条任务。 |
```

「说人话」表格列的写法见 **§14.1** 下「表格末列」；§3 观察指标表与非目标表 **SHOULD** 带该列。

### §4 落地选型与实施（已定稿）（MUST）

把「**维度上的取舍结论**」与「**怎么分批合进主线（代码落点与交付）**」钉在 spec 里；与 **§2** 调研材料相邻，便于 reviewer 先吃 **§4.1** 决策再对 **§4.2** 落地，再下钻 **§5** 协议与 **§6** One-Glance。

#### §4.0 章节编排

- **推荐**：与 [`read.md`](../../../../docs/architecture/tools/read.md) 标杆一致：**§1 术语 → §2 竞品调研 → §3 目标 → §4 已定稿选型与实施 → §5 协议 → §6 One-Glance → …**（文档内 `##` 编号可与 § 不同，**须**在文首或本节脚注说明对应关系）。

#### §4.1 落地选型决策表（MUST）

在 **§4** 内给出**至少一张**决策矩阵，**二选一主路径**（信息密度须等价，禁止只写一句「我们选了 X」）：

**主路径（推荐）：七列「维度上的取舍」矩阵**——下列七列列名与语义**须**一致（第三列 **`决策`**、末列 **`说人话`** 写法见 **§14.1**；与「入选理由」等列重复度过高时可合并单元格，但须保留**至少一句**可扫读的决策结论与一句口语扫读）：

```markdown
| 维度 | 关切 | 决策 | 取自 | 入选理由 | 未入选 + 拒因 | 说人话 |
| --- | --- | --- | --- | --- | --- | --- |
```

- **维度**：审计轴（如 R1–R6、安全 / 性能 / 协议等），一格一事。
- **关切**：本行要钉死的分叉问题（一句话可辩驳）。
- **决策**：**MUST** 每行 **一句**钉死本行**最终采纳结论**（宜「采用 X / 拒绝 Y」式裁决句）；**不**复述「入选理由 / 未入选 + 拒因」全文，**不**用口语替代审计列。业内外基线、竞品或备选路线的事实陈述放在 **§2** 横向表（可与 **§2** 互链，**禁止**在 §4.1 再堆一整张横向表代替本列）。
- **说人话**：**SHOULD** 每行 **一句**口语扫读（见 **§14.1** 表格末列）；**不**替代「决策」列的专业裁决表述。
- **取自**：证据来源。**MUST** 写成「`agent/仓库名 + 代码文件路径`（可附符号名）」；至少 1 条本仓代码证据 + 1 条外部 agent 证据。仅写“某报告/某竞品”不合格；若证据来自报告，仍须回指到报告中列出的**具体仓库文件**。
- **入选理由**：**MUST 同时包含两段语义**：`入选设计` + `入选理由（为什么）`（可含代价权衡、收益、边界）；只写“我们选 X”不合格。推荐句式：`设计：...；理由：...`。
- **未入选 + 拒因**：**MUST** 写明「哪些 agent 的哪些设计未入选」+「拒绝原因（为什么拒绝）」。至少 1 个并列弃案；若确无弃案，须显式写“无并列弃案”并给出证据链（PR/调研结论）。
- **代码落点、可观察交付物、阶段（PR / §4.2.x）**：**不在** §4.1 占列；**必须**在 **§4.2** 实施点表的 **`交付范围`（含交付物）** 与 **`主要代码落点`（含落地点）** 及 **`实施点`（PR/阶段命名）** 中写清，避免与 §4.1 重复堆叠同一句。

缺 **`决策`** 列或该列无裁决句 **不**视为缺 §4.1，但**新增或大面积改版**的架构文档宜每行补齐。

**第二路径：路线对比表**（与 [`search_files.md`](../../../../docs/architecture/tools/search_files.md) 中多列 A/B/C 路线对比同类）：多列对比若干候选路线，表下**必须**用一段话或短列表钉死 **最终采纳 = 哪一列 / 哪一组合**，并写明 **弃用列及原因**（等价于主路径的「未入选 + 拒因」）。**SHOULD** 为每行路线补 **一句**裁决结论（表内 **`决策`** 列或表下等价短句）。

工具 spec 中 **§2.3.x** 等**四列摘要表**（决策点 / 选择 / 现状/过去/其他 / 选择理由）**可**作为 §4.1 主表下的**可选扫读**，**不得**称为「形态 A」或替代七列主路径的 MUST 审计含义。

硬约束：

1. **一格一事 / 一行一路线**：每个可辩驳分叉独占表的一行（或独占路线列），禁止把整章结论揉成一行笼统话。
2. **「为何不是其他」可审计**：读者能回答「若不采纳本行**入选结论** / 若走并列弃案，会付出什么代价」（第二路径可由表下「最终决策」段承担）。
3. **确无并列弃案**的极少数单一路线：保留表头或等价标题，用一行写清 **「实行单一路线」** / 其他路线 **无** / 理由链 PR 或主方案 —— **禁止**删节整个决策矩阵。
4. **证据粒度可追溯**：`取自` 与 `未入选 + 拒因` 中出现的外部方案，均须落到「仓库/agent + 文件路径」级别（例如：`pi-mono/packages/ai/src/types.ts`），禁止只写产品名。

#### §4.2 实施点（已闭环）（MUST）

与 §4.1 **同章相邻**（前或后均可），给出实施点总表。下列 **五列**列名与语义**须**一致（末列「说人话」按 **§14.1** 为 **SHOULD**）：

```markdown
| 实施点 | 交付范围（含交付物） | 主要代码落点（含落地点） | 验收锚点（示例） | 说人话 |
|--------|----------------------|--------------------------|------------------|--------|
```

- **`实施点`**：PR / 看板命名或「单次合入」等阶段锚点；与 **§4.2.x** 小节标题一致或可映射。
- **`交付范围`**：**须**吸纳 §4.1 不再单列的 **交付物**（可观察产物、配置键、对外契约变更、文档节等）；可与 `实施点` 分工：此处写「交什么」。
- **`主要代码落点`**：**须**吸纳 §4.1 不再单列的 **落地点**（`*.rs` / 模块 / 关键符号）；此处写「落哪改哪」。
- **`验收锚点（示例）`**：测试函数名或见 **§11** 的指针；**禁止**「已测」而无指针。

末列「说人话」为 **SHOULD**（见 **§14.1**）。**禁止**与 §4.1 重复粘贴整段「入选理由」——§4.2 只保留**落地与验收**可执行表述；与 §4.1 行宜 **1:1 或一对多** 可映射，须在文首或表下脚注说明对应关系。

硬约束：

1. **SHOULD（强烈建议）**：表后按实施点拆小节（如 **`read.md` §4.2.1** 起），每小结配 **技术要点** + **ASCII**（或显式写「见图：本文调度 / 状态机章」），帮助把「表上一格」映射到调用栈与数据流。完整范例：[`read.md`](../../../../docs/architecture/tools/read.md) **§4.1–§4.2.x**；**同构另一例**：[`write.md`](../../../../docs/architecture/tools/write.md) 已定稿实施拆节（节号以该文档目录为准）。
2. **单行压缩（仅当单次合入、无子 PR 叙事）**：允许全表 **仅一行**——`实施点` 写 `单次合入` 或主 merge / tag；`验收锚点` 列**必须**含「见本文 **§11 测试矩阵**（§ 号随文档）」**或**直接列出 **≥1** 个测试函数名；**禁止**写「已测」而无指针。
3. **与 1 的边界（避免「含糊」）**：本条 **SHOULD** 的强度随**实施点行数**变化——表内 **≥2 行**（或多 PR、多交付边界）却**无**对应拆节、又无等价「见图 §x / §y」指针时，属可读性 **SHOULD** 未落实，review **可要求**作者补 **§4.2.1** 式小节；**不**自动等同 MUST 缺项。仅 **一行**且满足上条 **「单行压缩」** 时，**可不拆**子节。

### §5 协议（涉对外接口时 MUST）

用**字段表 + 子结构 + 调用样例**三件套描述：

| 字段 | JSON 类型 | 必填 | 默认值 | 适用场景 | 说明 | 说人话 |
|------|-----------|------|--------|----------|------|--------|
| ... | ... | ... | ... | ... | ... | ... |

要求：

1. 列出每一个字段（类型、必填、默认值、约束）；可选字段要说三态语义（缺省 / 显式 null / 显式值）。
2. 字段表 **SHOULD** 在末列加「说人话」（见 **§14.1**），便于非实现者扫读。
3. 子结构 / 嵌套 enum 单独画 ASCII 框图或代码块。
4. 至少一组**输入 + 输出**的调用样例（jsonc 注释形式可读性最佳）。
5. 标明**单一事实源**（哪一个 `*.rs` 是协议的最终定义）。

### §6 文件职责总览（One-Glance Map，MUST）

ASCII 框图，**不可省**。硬约束：

1. **覆盖完整**：方案改到的每个 `*.rs` 都有节点，缺一不可。
2. **箭头串联**：节点之间用 `│ + ▼` 或 `→` 标方向；自顶向下读一遍即复现整条链路。
3. **节点内要点**：列函数 / 类型 / 常量 / 关键行为，不允许只写文件名。
4. **挂独立 tests.rs**：节点内或下方注明配套 `[*/tests.rs]`（按 [`RUST_FILE_LINES_SPEC §A`](../coding/RUST_FILE_LINES_SPEC.md)）。
5. **图后阅读顺序**：紧跟 2–3 句 **「说人话」**（口语串链路），标清自顶向下谁调谁，与 **§14.1** 段落级要求一致。
6. **实施期偏离要标注**：如签名最终未改，节点上加 `【未改签名 / 依赖 Drop】` 之类显式标签，避免 stale 描述误导。

反例：

- ❌ 只画分层框（infra → core → api），不落到文件级。
- ❌ 节点只有文件名，没有函数 / 行为。
- ❌ 涉及 5 个文件，图里只画 3 个。

### §7 调度时序（SHOULD）

按需组合：

- **多列 ASCII 时序图**：如 [`search_files.md`](../../../../docs/architecture/tools/search_files.md)「调度时序」章，按「用户 / 入口 / 调度层 / 子进程」排列泳道；终端友好、易 diff。
- **Mermaid `sequenceDiagram autonumber`**：用于在 IDE 里渲染。
- **控制流伪码**：复杂嵌套循环 / 状态迁移用文字伪代码（如 [`agent-loop.md §13.3.2`](../../../../docs/architecture/agent-loop.md) 三层循环）。

每条事件 / 状态迁移必须显式标"何时发布、订阅方在哪"。

### §8 状态机（SHOULD）

用 ASCII 状态图 + 转移表二选一或并用：

```text
┌─────────┐ event_a  ┌─────────┐ event_b  ┌─────────┐
│  init   │─────────▶│ running │─────────▶│  done   │
└─────────┘          └────┬────┘          └─────────┘
                          │ cancel
                          ▼
                    ┌──────────┐
                    │  failed  │
                    └──────────┘
```

| 当前状态 | 事件 | 目标状态 | 副作用 | 说人话 |
|----------|------|----------|--------|--------|
| init | event_a | running | emit Started | 收到启动事件就进入跑起来的状态。 |
| running | cancel | failed | emit Failed{reason}, kill child | 用户取消就失败收尾并把子进程干掉。 |

### §9 配置与环境变量（SHOULD）

```markdown
| 变量 | 取值 | 含义 | 优先级 | 说人话 |
|------|------|------|--------|--------|
| `PI_SKIP_X` | `1` / `true` | 关闭 X | env（最高） | 环境变量一设就关掉某功能。 |
| `[section] x_enabled` | bool | 配置文件 | config | 配置文件里也能开关。 |
```

总则一律写明：**env > config > 默认**。

### §10 错误模型 / 截断 / 警告（SHOULD）

用 ASCII 框图列出**所有归一化结局**：哪些走 `Err`、哪些走 `truncated=true + warnings`、哪些静默兜底。避免读者脑补错误处理路径。

```text
正常 → warnings += "implementation=tier1"
分页截断 → truncated=true + next_offset
墙钟到点 → truncated=true + warning
容忍错误 → 空命中 + warning（不抛 Err）
致命错误 → Err（参数校验 / 权限拒绝 / IO 致命）
```

### §11 测试矩阵（MUST）

映射到**实际测试函数名**：

| 维度 | 用例 / 编号 | 状态 | 说人话 |
|------|-------------|------|--------|
| 单元 | `mod::tests::name`（n 用例） | ✅ 日期 | 模块内小测锁住细节。 |
| 集成 | `tests/foo_tests.rs::case_name` | ✅ | 黑盒或跨模块拼起来测。 |
| E2E | `E2E-XXX-NNN`（[`E2E_SCENARIO_LIBRARY.md`](../testing/E2E_SCENARIO_LIBRARY.md)） | PENDING | 用户能摸到的整条链路。 |
| 观察指标 | G1–G5 对应的锁死机制 | ✅ | §3 里吹的牛，这里用测试钉死。 |
| 文档 | 本文定稿 + 相邻方案同步 | ✅ | 字和代码别两张皮。 |

要求：

- §3 里每条目标在本表中能找到"锁死它的测试 / 机制"。
- 状态列只允许 `✅ 日期` / `PENDING` / `阻塞于 X`，禁止留空或写"待补"。

### §12 风险与应对（MUST）

```markdown
| 风险 | 影响 | 应对（具体动作） | 说人话 |
|------|------|--------------------|--------|
| 大仓库 Tier2 超时 | 中 | 10 s 墙钟 + truncated=true + warning；env 可调 | 慢就截断返回，别无限等。 |
```

应对必须落到具体动作（"加 `tokio::time::timeout(2s)`"、"`fsync` 后再返回"）。至少覆盖：并发 / 信号 / 兼容性 / 回滚 / schema 破坏 / panic 隔离 / 资源泄漏 中相关项。

### §13 历史决策 / 跨文档修订（SHOULD）

- 用删除线 + "→ 否：理由"形式列出**已被取代**的方案，让 reviewer 一眼看到设计是怎么演化到当前的。
- 若本方案改写了相邻方案文档某段语义，单开"跨文档修订"小节登记修订意图（即使本 PR 暂未同步改）。

## 14. 写作风格（与 [`MODULE_README_SPEC.md`](MODULE_README_SPEC.md) 共用）

| 项 | 要求 |
|----|------|
| 精准引用 | 文件 / 符号 / 行号都尽量带路径 |
| 术语一致 | 与术语表保持，不允许同义词混用 |
| 时间点钉死 | "LLM 回复后" 一类模糊词必须用一句话约定指代位置 |
| Why 优先 | 先说"为什么"，再说"怎么做"，再说"做了什么" |
| ASCII 优先 | 终端友好、易 diff；复杂时序可补 mermaid |
| 表格优先 | 选型 / 状态转移 / 风险 / 验收 / 配置等结构化内容用表格 |
| **说人话** | 见 **§14.1**（**先专业、后口语**：段落辅助 + 表格末列）；与 [`Constitution.md`](../../Constitution.md) **二.10** 同向 |
| No-Stale | 实施期调整须就地更新文档，并标 `【未改签名 / 依赖 Drop】` 等显式标签；禁止留"原计划/现落地"两份真相 |
| 短句收尾 | 篇幅 > 300 行的方案末尾补一段"一句话总结"（参考 [`interrupt-and-cancellation.md §4.6`](../../../../docs/architecture/interrupt-and-cancellation.md) 末尾） |

### 14.1 「说人话」辅助理解（SHOULD）

落实 [`Constitution.md`](../../Constitution.md) **二.10**。**总顺序**：**先**写清、写准专业技术内容（定义、契约、步骤、边界、可归一化结局）；**再**用口语化文字帮助读者扫读。**禁止**用「说人话」替代必须的技术表述。

#### 段落或小节末尾（含图后「阅读顺序」）

- 在每个**主要**技术小节、或整段 ASCII / Mermaid 技术描述之后，**SHOULD** 可紧跟 **2–5 句**口语串意图；可用连续段落，或小标题 **`说人话`** / **`阅读顺序（说人话）`**。
- **One-Glance Map** 图后的「阅读顺序」段归入本类（与 **§6「文件职责总览」** 硬约束 5 一致）。
- **豁免**：纯目录、纯索引、过渡句已足够短、或该块正文本身已是低密度说明时，不强制再加。

#### §4.1「决策」列与末列「说人话」

**§4.1** 七列维度取舍矩阵：**第三列**列名固定为 **`决策`**（见上文 **§4.1** 语义），**末列**固定为 **`说人话`**。每行 **决策** 列 **一句**裁决结论：钉死本行最终采纳（宜「采用 … / 拒绝 …」式）；**不**复述「入选理由 / 未入选 + 拒因」全文，**不**用口语替代审计列。基线/对标事实写在 **§2**，**不**占用 **决策** 列。**说人话** 列每行 **一句**口语扫读，**不**替代 **决策** 列。

#### 表格末列「说人话」（§4.1 以外）

在信息密度高的 Markdown 表末追加一列，列名固定为 **`说人话`**，与 [`architecture/tools/edit.md`](../../../../docs/architecture/tools/edit.md) 等标杆一致。**§4.1** 矩阵同时含第三列 **`决策`** 与末列 **`说人话`**（分工见上节）。

**何时 SHOULD 加**

- 原则 / 观察指标（G 表）、非目标、竞品横向、现状基线、**§4.2** 实施点总表、按 `#` 编号的实施落地点小节表、协议字段表、状态机转移表、配置表、错误归一化表、测试矩阵、风险表、历史决策表、附录「旧节号 ↔ 新版」对照表等（**§4.1** 用 **`决策`** + 末列 **`说人话`**，见上节）。

**写什么**

- 每行 **一句**口语化中文（可含少量术语），供扫读与 onboarding。
- **不**复述同行技术列的完整定义，**不**替代「观察指标 / 字段说明 / 为何不是其他」正文列；与 **§1 术语表「语义（大白话）」** 可并存：术语表以三件套为主；若表内已有「大白话」列，「说人话」宜更短，作页内速记。

**豁免（可省略该列）**

- 纯路径 / 符号索引、或列数 ≤2 且每格已是一句完整大白话的极短表。
- **补充型架构文档**（不承担完整工具类骨架者，范例见 **§5**、[`interrupt-and-cancellation.md`](../../../../docs/architecture/interrupt-and-cancellation.md) **§1.3**）：若主方案已含完整「决策 / 说人话」列且本文不新增可辩驳分叉，表格末列可不重复；若本文新增 **§4.1** 决策行，**SHOULD** 为新增行补「决策」；其他表补「说人话」。

**与 MUST 的关系**

- 缺「说人话」列或缺段落辅助 **不**构成缺 §4.1 / §4.2 / §11 等 MUST 项；属可读性 **SHOULD**。新增或大面积改版架构文档时优先补齐。

## 6. 反例汇总

- ❌ §3 目标用散文："本方案旨在改善搜索体验"——必须翻成观察指标表。
- ❌ 协议节漏掉默认值或必填标记。
- ❌ One-Glance Map 只画分层，不落到文件级。
- ❌ 状态机用一段散文描述："收到事件后会进入 running 状态" —— 必须配 ASCII 状态图 + 转移表。
- ❌ 风险节写"注意一下并发问题"——必须到"加 `tokio::time::timeout(2s)`"这种动作粒度。
- ❌ 验收节"已通过测试"——必须列测试函数名 + 状态。
- ❌ 落地选型章只有散文或只列「我们用了 X」——缺 **§4.1 落地选型决策表**（七列主路径或第二路径路线对比）或 **§4.2 实施点（已闭环）**（且无 **§4.0** 编排说明，亦无 **§1.3** 式对 §4.1/§4.2 的等效映射与回链）。
