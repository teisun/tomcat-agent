| **来源** | 2026-06-02：基于正式多 LLM 产品化架构 spec 立项。Tomcat 当前 `OpenAiProvider` 在构造时锁定 `base_url` / `api_key` / `default_model`，运行时只会改请求体里的 `model` 字符串，无法在同一会话稳定切换 OpenAI / DeepSeek；同时 chat 缺 `/model` 入口、CLI 不持续显示当前会话模型、`tomcat init` 仍是单 provider 心智。 |

**目标**：按架构文档落地 **model-first** 的多 LLM 产品化基线：引入 `ModelCatalog + LlmResolver + AuthStore(最小闭环)`，让用户能在聊天态查询/切换模型，所选模型持久化到会话并在程序重入后继续生效，同时初始化向导按所选 model 推导并配置对应凭证。

**范围**：

- 落地架构文档 `W1-1` ~ `W1-5`，并补上 `W2-1` 的最小多 key 闭环（provider → `<PROVIDER>_API_KEY`）。
- 保持 `resolve_llm`、`thinking_policy`、reasoning continuity 主骨架不变；路由按 `api`，`provider` 只用于 auth / catalog / 展示。
- 不做 `W2-2` FallbackChain / unhealthy cache、`W2-3` usage/metrics 聚合、`W3-x` 第二家非 OpenAI 形 provider / OAuth / TUI picker。

**子项**（与架构文档 `§4.1` 决策表、`§4.2` 实施点一致）：

- [x] **catalog**（`W1-1` / `R1/R2/R7`）：实现 `ModelCatalog` / `ModelEntry`、内置常用模型表、`~/.tomcat/models.toml` 合并、显式选模 miss 报错与 legacy fallback
- [x] **resolver-scenes**（`W1-2` / `R3/R4`）：实现 `LlmScene` / `ResolvedCall` / per-call Resolver，把主对话 `effective_model` 逻辑并入 resolver，并补 `vision_model` / `title_model` 等 scene 键
- [x] **authstore-minimum**（`W2-1` / `R5`）：实现 provider → `<PROVIDER>_API_KEY` 映射、缺失可读错误、保留现有 `api_key_env` 兜底
- [x] **capability-guard**（`W1-3` / `R7`）：基于 catalog `capabilities` 在调用前拦截 vision / files 等不支持组合，并给出结构化引导错误
- [x] **chat-model-ux**（`W1-4` / `R8`）：新增 `/model current|list|use`；`/model use <id>` 持久化写入 `SessionEntry.model_override` / `sessions.json`，落 `model_change` transcript，并让 CLI 横幅与输入 prompt 持续显示当前对话 model
- [x] **init-model-wizard**（`W1-5` / `R9`）：扩展 `tomcat init` 为 model-first 交互式向导；先选 `default_model`，再按 catalog 推导 `provider/api` 并提示写入所需 key
- [x] **tests-and-doc-sync**：补齐本卡涉及的 `§11` 测试锚点，并同步相邻文档交叉引用与用户可见帮助文案

**依赖**：

- 现有 `resolve_llm` 注册表、`OpenAiProvider` / `OpenAiResponsesProvider`、`SessionEntry.model_override` 与 `model_change` 最小链路为前置
- [`llm-multiprovider-integration.md`](../../../architecture/llm-multiprovider-integration.md) 与 [`llm-openai-deepseek-reasoning-continuity.md`](../../../architecture/llm-openai-deepseek-reasoning-continuity.md) 为前置事实来源
- 无新增开放任务硬阻塞；建议在已合入 `T2-P1-010` 推理续传成果的基础上推进

**被依赖**：后续 TUI model picker、FallbackChain/健康度、按 model 聚合计量等波次能力将复用本卡产物。

**协作接口**：

- `src/core/llm/{catalog.rs,resolver.rs,auth.rs,registry.rs,token_usage.rs}`
- `src/api/chat/{context.rs,prompt.rs,run_loop/mod.rs}`
- `src/api/chat/commands/{parse.rs,cmd_help.rs,cmd_model.rs}`
- `src/api/cli/{init.rs,init_model_wizard.rs}`
- `src/core/session/{store.rs,transcript.rs}`
- `src/infra/config/types/{llm.rs,context.rs}`

**验收标准**：

- 对齐架构文档 `§11` 的核心锚点：`core::llm::catalog::tests::{resolve_known_model,merge_user_override,missing_explicit_model_errors,legacy_fallback_entry}`、`core::llm::resolver::tests::{scene_fallback_to_main,override_priority,reject_vision_on_text_model}`、`core::llm::auth::tests::{per_provider_env,missing_key_message}`
- `/model current|list|use` 可在 `tomcat chat` 中工作；`/model use <id>` 后 CLI 横幅 / prompt 能显示当前对话模型，重启程序并恢复同一 session 后仍保持该 model
- `[llm].default_model` 仍只作为“未显式选过模型的会话”默认值，不与已持久化的 `SessionEntry.model_override` 冲突
- `tomcat init` 能基于 catalog 选择 `default_model`，并按所选模型正确提示 / 写入 `OPENAI_API_KEY`、`DEEPSEEK_API_KEY` 等 provider 凭证
- 未配置 catalog / scene / 多 key 时保持现状兼容，现有相关集成测试不回归（G7）
- `cargo fmt --check`、`cargo clippy --all-targets -- -D warnings`、相关 `cargo test` / 集成测试全绿

---

**备注**：本卡是“多 LLM 产品化”的 **Wave 1 基线任务**；若实施中需要拆 PR，可按 `catalog → resolver/auth → /model + CLI → init` 的顺序切分，但验收以本卡整体闭环为准。
