# 开发计划规范

本文定义工程师 Agent 写开发计划时的统一格式与质量要求。
目标只有三个：**可评审、可执行、可验收**。不再区分“普通计划 / 大任务计划 / bug 计划”多套模板，统一采用**一套 5 章主干结构**；如任务确有需要，可在此基础上追加其他章节。

与 [Constitution.md](../../openspec/specs/Constitution.md) 和 [ARCHITECTURE_SPEC.md](../../openspec/specs/guides/workflow/ARCHITECTURE_SPEC.md) 一致，计划必须遵守**先专业、后说人话**：先把路径、接口、边界、顺序、测试写清，再用口语化中文帮助读者扫读，不能只写口语废话。

---

## 一、统一格式（强制）

说人话：先把共同骨架定死，后面大家讨论的才是同一种计划，而不是每个人各写各的格式。

每份计划**至少**按下列形式输出：

```markdown
---
name: <计划标题>
overview: <一句话说明要解决什么问题>
isProject: false
todos:
  - id: <todo-id>
    content: <动作>
    status: pending
---

# <计划标题>

## 1. 背景
## 2. 竞品 / 方案对比
## 3. 评审 QA
## 4. 决策表
## 5. 测试方案与验收
```

这 5 章是**强制最小骨架**，不是最大上限。可以按任务复杂度追加 `0. 执行摘要`、`6. 历史补充 / Todo 对照 / 附录` 等章节，但**不能**删掉、替换或跳过这 5 章。

| 部分 | 要求 | 说人话 |
| :--- | :--- | :--- |
| `frontmatter` | `name`、`overview`、`todos`、`isProject` 必填；`todos` 里的 id 必须能在正文里找到落点，且**必须显式覆盖本任务的研发流程关键节点**（如认领 / 读规范 / 建分支 / 开发 / 测试 / status / commit / push / 交接）与**决策表全部已拍板事项**。 | 别把 todo 藏在脑子里；不仅要能顺着 id 找到实施点，还要保证真正会发生的流程和拍板内容没有漏。 |
| 5 章正文 | `1` 到 `5` 这 5 章是强制主干；可以在章内继续分小节，也可以额外追加别的章节，但不要用别的总结构替掉这 5 章。 | 以后大家先按这 5 章找主干；真有历史补充或附录，再往后加。 |
| 章节引导 | **每一个章节、每一个小结、每一张表**都必须显式出现 `说人话`：章节至少有 1 段；小结标题建议直接写成 `（说人话）` 或 `阅读顺序（说人话）`；表格优先单独设 `说人话` 列，若表形态确实不适合加列，表后也必须紧跟一段 `说人话`。 | 计划不是写给作者自己看的，扫一眼要能知道“这章在解决什么、这张表到底想说明什么”。 |

**禁止事项**：

- 不要把计划写成第二份规格全书。
- 不要写“方向对、后面再说”这类空话，必须落到背景、结论或方案。
- 不要把 bug 修复单独写成另一套格式；仍然用这 5 章，只是在 `背景`、`评审 QA`、`决策表` 里把问题讲透。

## 二、5 章怎么写

说人话：下面不是在教“标题怎么排版”，而是在规定每一章到底要承担什么职责，避免看起来很完整、其实没有信息密度。

| 章节 | 目的 | 最少要写什么 | 说人话 |
| :--- | :--- | :--- | :--- |
| `1. 背景` | 交代现状、差距、边界 | 当前基线、问题/目标、单一事实来源、阶段边界、已知约束 | 先说明“为什么现在要做这件事”，别上来就改方案。 |
| `2. 竞品 / 方案对比` | 把“为什么选这个”讲明白 | 外部竞品或内部 A/B 方案对比、取舍依据、明确不选什么 | 不只是列方案名，要说清楚为什么借、为什么不借。 |
| `3. 评审 QA` | 提前回答评审会问的问题 | 每个问题的专业描述、结论、说人话 | 把容易卡住实施的疑问提前打掉。 |
| `4. 决策表` | 把已拍板事项收口 | 决策表、实施点、小结、必要时的文件职责图 | 真正的落地动作在这里，不要只有原则没有落点。 |
| `5. 测试方案与验收` | 说明怎么证明做完了 | 测试层次、覆盖矩阵、验收口径、不纳入本期的边界 | 说清楚“怎么验”比说“应该没问题”有用得多。 |

### 2.1 `1. 背景`

说人话：背景这章就是先交代“为什么现在要做、现在差在哪、这次边界到哪”，别一上来就冲到方案。

`背景` 章至少覆盖以下内容：

1. **当前基线 / 现状与差距**：当前行为、目标行为、差距在哪里。
2. **单一事实来源**：若依赖架构文档、task 卡或现有协议，必须点名写出权威文档路径。
3. **阶段边界 / 非目标**：若本期只做 Phase 1、只改文档、不改 `src/`、或把部分动作推到后续阶段，必须写清楚。
4. **为什么现在做**：风险、收益、用户影响、后续工作的前置关系。

若计划是 **bug 修复类**，`背景` 章还必须补齐：

| 要素 | 要求 | 说人话 |
| :--- | :--- | :--- |
| 现象 | 给出用户可见异常和具体数据 | 不能只写“好像有问题”。 |
| 期望 | 正确情况下应该是什么 | 不写期望，就不知道修没修对。 |
| 量化偏差 | 异常值和期望值差多少 | 差 10B 和差 500KB 完全不是一个问题。 |
| 对比组 | 至少一组正常对照 | 有对照，才能判断问题是偶发还是机制性。 |

若 bug 涉及**运行时状态不一致 / 水位漂移 / 窗口错位**等问题，建议在 `背景` 内追加**逐步场景**（Step-by-Step）说明“在哪一步开始脱节”；需要时可用“两本账”写法把数字账和消息账并排写清。

### 2.2 `2. 竞品 / 方案对比`

说人话：这章不是为了显得做过调研，而是为了说明“为什么选这个，不选另外那些”。

这一章不要求一定有“外部竞品”。没有外部竞品时，也必须做**内部方案对比**。

推荐表头：

| 方案 | 核心做法 | 优点 | 风险 / 成本 | 结论 | 说人话 |
| :--- | :--- | :--- | :--- | :--- | :--- |

要求：

- 至少比较 **2 个可行方案**；不能只写“最终方案”。
- 结论必须回扣本任务约束：性能、复杂度、兼容成本、阶段边界、用户收益、可测试性。
- 若借鉴竞品，写清楚**借哪部分**、**不借哪部分**。
- 若问题是纯 bug 修复，也要比较“治标 vs 治本”或“局部补丁 vs 根因修复”。

### 2.3 `3. 评审 QA`

说人话：把评审时最容易卡住实现的问题提前答掉，比事后在评论区来回解释省很多时间。

这一章不是 FAQ 堆料，而是把**真正会卡住实现或评审**的问题提前说透。

每个 QA 建议使用固定结构：

```markdown
### Q1. <问题>

**专业描述**
<问题的技术上下文、约束、边界>

**结论**
<已拍板结论；若有条件适用也写清>

**说人话**
<一句到三句，把为什么这么定讲明白>
```

要求：

- QA 必须服务于决策，不能只是重复背景。
- 对跨时序因果链，鼓励直接写“你可能会问……我们的结论是……”，主动消灭歧义。
- 若某个问题尚未拍板，就不要伪装成结论；应在 `决策表` 标成待确认项。

### 2.4 `4. 决策表`

说人话：前面可以讨论，这一章开始就要收口，明确什么已经拍板、怎么落地。

这是全篇最重要的一章。推荐至少使用下列表头：

| 议题 | 主要参考 | 决策 | 落地方式 | 说人话 |
| :--- | :--- | :--- | :--- | :--- |

要求：

- 只写**已拍板**或**明确待确认**的事项，不能把未想清楚的问题伪装成“默认如此”。
- `落地方式` 必须写到文件 / 模块 / 数据结构 / 调用点，不要只写概念。
- 每一行都要能追到具体实施动作或明确的“不做”边界。

#### 4.1 实施点 + 小结

说人话：决策表讲“定了什么”，这一小节讲“具体从哪里下手、按什么顺序落地”。

决策表之后必须紧跟 `实施点 + 小结`。这部分用于把决策翻译成真正可执行的改动。

每个实施点都要写清：

| 项 | 要求 | 说人话 |
| :--- | :--- | :--- |
| 文件与模块 | 改哪些源码、配置、文档；依赖哪些规格 | 先点名改哪里，读者才知道从哪下手。 |
| 实现思路 | 关键数据结构、调用链、触发时机；跨 async 边界必须写清约束 | 谁调谁、何时调、哪里不能阻塞，要先说清。 |
| 接口影响 | 复用哪些已有接口，新增或变更哪些接口 | 避免做到一半才发现签名不够用。 |
| 测试要点 | 正常路径、边界、异常、回归风险 | 不是“最后补个测试”，而是先设计怎么证伪。 |
| todo 对应 | 标出对应 `todo id` | 正文和 frontmatter 要能双向对上。 |

补充要求：

- **实施顺序与依赖关系不再单独开总章**。如果任务有明显先后关系，请在实施点顺序里直接表达；中大型任务可在本小节末尾补一张 ASCII 关键路径图。
- 若任务跨多个 Phase，请在这里顺手写清 `本期做什么 / 本期不做什么`，不要散落到别处。
- 若某个修复对应具体根因，必须写明“修哪条根因”；能治本就不要只给治标补丁。
- 若改动了数据的生产方式，实施点里要补一个**下游兼容性检查**，点名谁会消费这份数据。

#### 4.2 文件职责总览（One-Glance Map，条件触发 MUST）

说人话：一旦任务跨多个业务文件，只靠文字很容易看丢链路，所以这里必须给读者一张“从哪进、往哪流、最后落到哪”的总览图。

当本计划的关键改动涉及 **≥ 2 个 `*.rs` 业务源文件**（含独立 `tests.rs`）时，必须在 `实施点 + 小结` 内提供一张 One-Glance Map。

硬约束沿用 [ARCHITECTURE_SPEC.md](../../openspec/specs/guides/workflow/ARCHITECTURE_SPEC.md) 中同名要求：

1. 节点间必须用 `│` + `▼` 或 `→` 标方向。
2. 关键改动涉及的每个 `*.rs` 都必须有节点。
3. 节点内必须写函数 / 类型 / 常量 / 关键行为，不能只写文件名。
4. 要同时标出配套独立 `tests/`。
5. 图后要跟 2 到 3 句**阅读顺序（说人话）**。
6. 若实施期发现原计划图有偏差，须在图上就地标注 `【未改签名】`、`【保留旧出口】` 等显式标签。

仅 1 个业务源文件的小改动可写：`不适用：仅改 <文件>`。

### 2.5 `5. 测试方案与验收`

说人话：这章不是写“应该没问题”，而是写“怎么证明它真的没问题、做到什么程度才算交付”。

这一章至少包括：

1. **测试层次**：单测 / 集成 / E2E / 手工验证分别覆盖什么。
2. **覆盖矩阵**：正常路径、边界、异常、回归点各由什么测试承担。
3. **验收标准**：fmt、clippy、测试、场景验证、文档更新等完成口径。
4. **不纳入本期验收**：明确写清哪些风险暂不处理，避免“默认都包了”。

若任务涉及**用户可见行为**、CLI、对话流、插件、协议或 P0/P1 用户故事，必须补齐：

- 对应的 [E2E_SCENARIO_LIBRARY.md](../../openspec/specs/guides/testing/E2E_SCENARIO_LIBRARY.md) 场景或待新增项；
- `tests/` 下的集成测试落点；
- 必要时 `tests/cli_tests.rs` / `tests/wasmedge_e2e_tests.rs` 的覆盖点；
- 交付前全量门禁按 [INTEGRATION_MERGE_AND_ACCEPTANCE.md](../INTEGRATION_MERGE_AND_ACCEPTANCE.md) 执行。

纯内部重构可以写“不适用”，但必须补一句理由。

## 三、表达与排版要求

说人话：这一节是把“怎么写得既专业又好读”讲清楚，避免内容是对的，但读起来像一堵墙。

| 规则 | 要求 | 说人话 |
| :--- | :--- | :--- |
| 先专业、后口语 | 先写术语、路径、契约、边界，再写 `说人话` | 先把事说准，再把话说顺。 |
| 章节小结 | 每章至少有一段显式 `说人话`；章内小结标题最好直接带 `（说人话）` | 让读者知道这章看完该带走什么。 |
| 表格口语列 | 每张表都必须显式有 `说人话`：优先单独一列；若表形态不适合加列，表后必须紧跟一段 `说人话` 小结 | 信息密度高的时候，给读者一个快速翻译。 |
| 小结显式标记 | `阅读顺序（说人话）`、`实施小结（说人话）`、`验收小结（说人话）` 这类写法应优先使用 | 别让读者猜哪句是总结、哪句是正文。 |
| 不写空话 | 禁止“后面再优化”“大致如此”这类无落点句子 | 每句话都要能回答“所以呢”。 |
| 路径与符号 | 文件路径、模块、函数、配置项都要点名 | 计划能直接指导改代码，而不是只讲理念。 |
| 控制篇幅 | 只保留与本期决策、实施、验收有关的内容 | 计划不是把所有背景资料重新抄一遍。 |

关于 `todos`：

- `frontmatter.todos` 是追踪源；
- 正文中的实施点、决策或 QA 必须能回到对应 `todo id`；
- `todos` 必须显式反映本任务的**研发流程**，至少覆盖真正会执行的关键节点；若顺序受流程约束（如先开分支才能开发、先登记 test-group 才能跑全量集成），要在 todo 顺序里表达出来；
- `todos` 必须覆盖 `决策表` 的**全部已拍板项**；可以一行决策对应一个 todo，不能出现「决策表写了、todos 没落点」；
- 若看板子项编号与计划实施点编号不一致，须补一张“看板项 ↔ 实施点 / todo”映射表。

## 四、输出前自检（必做）

说人话：交出去前按这张表再扫一遍，目的是发现“结构看起来齐了，但读者还是看不懂”的漏口。

- [ ] 已使用统一 `frontmatter + 5 章主干` 结构；如追加章节，也未替代这 5 章
- [ ] 每一章、每一个小结、每一张表都显式出现了 `说人话`
- [ ] `todos` 与正文实施点可双向对应
- [ ] `todos` 已显式覆盖本任务研发流程关键节点，且前后顺序符合实际执行约束
- [ ] `决策表` 的全部已拍板事项都已映射到 `todos`，无遗漏 / 无悬空决策
- [ ] `背景` 写清了现状、目标、边界、单一事实来源
- [ ] 若是 bug 修复，已写现象 / 期望 / 量化偏差 / 对比组
- [ ] `竞品 / 方案对比` 至少比较了两个可行方案
- [ ] `评审 QA` 的每个问题都包含专业描述、结论、说人话
- [ ] `决策表` 每一行都能追到具体落地动作或明确的不做边界
- [ ] `实施点 + 小结` 已覆盖文件、思路、接口、测试、todo 对应
- [ ] 若有明显先后依赖，已在实施点顺序或 ASCII 关键路径图中表达
- [ ] 若涉及 ≥ 2 个 `*.rs` 业务源文件，已提供 One-Glance Map；否则已写不适用
- [ ] `测试方案与验收` 已覆盖正常 / 边界 / 异常 / 回归
- [ ] 用户可见任务已写集成 / E2E / 场景库落点；纯内部任务已写不适用理由
- [ ] 表格和章节的小结没有流于空话，真正解释了上下文与结论

说人话：这一节告诉你要从哪份样板开始抄骨架、从哪份案例看复杂任务怎么写实，不用每次从零发明格式。

| 文档 | 用途 | 说人话 |
| :--- | :--- | :--- |
| [PLAN_SKELETON.md](./PLAN_SKELETON.md) | 统一 5 章骨架，复制后填空 | 新计划先从这里起步，别再手搓旧结构。 |
| [PLAN_EXAMPLE_CURRENT_TAIL_AGGREGATE_GUARD.md](./PLAN_EXAMPLE_CURRENT_TAIL_AGGREGATE_GUARD.md) | 完整样板：新 5 章结构 + QA + 决策表 + 实施点 + One-Glance Map | 想看“复杂任务怎么把新规范写实”时，优先参考这份。 |
