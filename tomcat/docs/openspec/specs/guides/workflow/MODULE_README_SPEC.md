# Module README 规范

> 适用：`src/**/README.md`（与代码目录一一对应的模块说明）。
> 上位规范：[`Constitution.md`](../../Constitution.md) · [`DOCUMENTATION_GUIDE.md`](DOCUMENTATION_GUIDE.md)。

## 1. 何时写

- 任意一级 `src/` 子目录默认应有一份 `README.md`；新增子模块时必须同步建。
- 与代码改动同步更新；**不替代**用户指南、组内分享、status 进度文档。
- 若某项设计点同时出现在本 README 与 `docs/architecture/` 下的方案文档，**以方案文档为权威**，README 加回链不重复内容。

## 2. 章节模板

```markdown
# [模块名] 模块说明

## 1. 概述
- 职责：一句话写清模块解决的问题。
- 所在层级：基础设施层 / 核心逻辑层 / 接入层。
- 核心文件：`src/<path>/...`。
- ASCII 缩略图：标本模块在系统栈的位置；与 `src/README.md` 总览图对照，避免重复画整张系统图。

## 2. 设计要点
- 设计模式 / 关键权衡（Why vs What）。
- 并发模型：是否 `Send + Sync`；锁 / 通道 / 原子的取舍。
- 与邻接模块的边界：消费什么、提供什么。

## 3. 核心 API 与数据结构
- 用代码块列 trait / struct 签名；带关键字段注释。
- 错误变体：列出本模块特有的 `AppError` 取值。

## 4. 配置项
- 环境变量、`tomcat.config.toml` 字段、默认值；与 [`infra/config/types.rs`](../../../../src/infra/config/types.rs) 对齐。

## 5. 交互流程
- 典型调用链：初始化 → 注册 → 触发 → 清理；建议附一段 ASCII 时序。

## 6. 用法示例
- 至少一个"快速上手"代码块；可指向集成测试文件。

## 7. 验收
- 关键测试函数名、覆盖率门槛、性能指标。
```

## 3. 写作要点

| 项 | 要求 |
|----|------|
| 精准引用 | 文件 / 符号要带路径，跨目录用 Markdown 相对链接 |
| 术语一致 | 全文用术语表里钉死的那个词（同义词不允许混用） |
| Why 优先 | "为什么这么实现"重于"实现了什么"；常识性技巧不展开 |
| ASCII 优先 | 概览图首选 ASCII（易 diff、终端友好）；复杂时序可补 mermaid |
| 表格优先 | 配置项 / 错误变体 / 测试映射等结构化内容用表格，不用散文 |
| 说人话 | 主要技术节后 **MAY** 跟短段口语辅助；多列对照表 **MAY** 末列 **`说人话`**，规则同 [`ARCHITECTURE_SPEC.md`](ARCHITECTURE_SPEC.md) **§14.1**（A 类非强制） |
| 与代码同步 | 改动 API 必须同步本 README；评审禁止留"旧/新"两份真相 |

## 4. 反例

- ❌ 概述只复制 crate 文档第一段，没说本模块在系统中的位置。
- ❌ "API 与数据结构"一节列了一个 `Foo`，但实际公开的是 `FooBuilder` + `FooHandle`。
- ❌ 配置项没写默认值。
- ❌ "验收"写"已通过测试"，没列测试函数名。

## 5. 顶层索引

模块 README 需在 [`src/README.md`](../../../../src/README.md) 的目录树中可见；新增模块时同时更新该索引。
