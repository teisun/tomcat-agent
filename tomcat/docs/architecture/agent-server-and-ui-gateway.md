# Agent Server 与 UI Gateway：把 Tomcat CLI 能力暴露给 VSCode / 桌面 GUI 的进程边界方案

> 适用范围：`tomcat` 把宿主核心能力（`AgentLoop` 一次完整 run、流式事件、工具时序、审批/提问、会话控制）暴露给**进程外 UI**（VSCode 插件、Electron/Tauri 桌面 GUI、Web、远程客户端）的传输与协议设计。
> 上位规范：`[ARCHITECTURE_SPEC.md](../openspec/specs/guides/workflow/ARCHITECTURE_SPEC.md)`。本文 `## 1`–`## 10` 与规范 §1–§10 一一对应；文首「方案导图集」置于 `## 1` 之前、不占用 § 编号。
> 关联文档：`[interaction-layer.md](./interaction-layer.md)`（交互层入口页，§3 已声明「为未来 Web / TUI / IPC 预留一致接口形态」）、`[llm-stream-events-cli-pipeline.md](./llm-stream-events-cli-pipeline.md)`（`StreamEvent → AgentEvent → EventBus → CLI` 现状链路）、`[agent-loop.md](./agent-loop.md)`、`[multi-agent.md](./multi-agent.md)`。
>
> 本文回答四件事：
>
> 1. **做 VSCode 插件 / 桌面 GUI 前，是否要先做「网关 gateway」？** —— 不是先做网络服务，而是先做**进程边界协议**；首发传输是 **stdio NDJSON**（子进程模式），网络网关（WS/HTTP daemon）是可选的 Phase 2 泛化。
> 2. **其他 agent 用什么方式把能力暴露给 UI？gateway / CLI / SSE？** —— 见 **§2** 横向调研：主流是「**行分隔 JSON over stdio 子进程**」，部分升级到「**一个 dispatcher + 多传输（stdio + WebSocket）**」；**SSE 几乎不是**主链路（仅 openclaw 用于历史回放、cc-fork 用于云端、hermes 用于 OpenAI 兼容 API）。
> 3. **能不能像 Cursor / Copilot / Codex 那样，一个进程同时跑多个会话（多 tab 并发）？本期就要做。** —— 见 **§2.5** 横向调研 + **§3.3** 落地方案：**能，而且本期做**。Tomcat 核心（`AgentLoop` 多实例、`EventBus` 已按 `sessionId` 打信封、`AgentRegistry` 已做进程级登记 + 级联中止 + 并发上限）已具备多会话并发底座；缺的是 `multi-agent.md` 维度A 规划但**尚未实现**的 `ChatContextRegistry` 与 serve 传输层的多路复用（`sessionId` 命令路由 + per-session run task + per-session interrupt + 单写者按 `sessionId` 公平 demux）。这正是 **codex `app-server` 的 `ThreadManager` 模型**（单进程多 thread、跨 thread 真并发、同 thread 串行），与 Tomcat 同语言、可直接对标。
> 4. **Tomcat 应该怎么落地？** —— 见 **§3**：复用现有 `AgentEvent` wire（已与 pi-mono 对齐）+ 单写者 drain + 正交 control 通道 + 复用 `EventBusAskQuestionPanel` 做审批回环 + `**ChatContextRegistry` 多会话并发**；新增 `tomcat serve` 子命令，不动 `AgentLoop` 核心。

## 文首导读：方案导图集

一句话先抓住结论：**「gateway」不是首先要做的网络服务器，而是一层「把 EventBus 事件 ↔ 进程外 wire」翻译 + 会话拥有 + 命令分发的 dispatcher；它的第一个传输应该是 stdio（子进程），WebSocket/HTTP daemon 只是后续把同一 dispatcher 多挂一个传输。本期这层 dispatcher 直接做成「单进程多会话并发」：内部一张 `ChatContextRegistry`（`sessionId → Arc<ChatContext>`），每个会话各自一个 `AgentLoop` run task、各自一个中断信号，事件靠既有 `WireEnvelope.sessionId` demux——这正是 codex `app-server` 的 `ThreadManager` 模型。**

#### 连接级三件套（图中 dispatcher 第一行）

> 这三项都是**一条连接共享一次**的基础设施，不是 per-session 职责；`ChatContextRegistry` 与多 session 并发挂在这层**下面**。顺序是：连接先握手 → transport 统一读写 → 多个 session 各自跑 → 所有下行经单写者发出。


| 项                      | 专业描述                                                                                                                                                          | 说人话                                    | 为何这样设计                                                                                                                                                             | 必要性                                                                                            |
| ---------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ---------------------------------------------------------------------------------------------- |
| `**initialize` 握手**    | 连接建立后的 bootstrap / capability negotiation：客户端发 `initialize`，服务端回 `protocolVersion`、`capabilities`（及可选默认 `sessionId`），双方进入 `ready` 后才接受 `prompt` 等业务命令。        | 先打招呼，确认「咱俩说的是不是同一套协议、各自会哪些动作」。         | Client 与 server 会独立演进；握手一次性钉死版本与能力位（多 session、`ask_question`、`interrupt` 等），避免两端各自硬编码假设。                                                                               | 无握手则版本漂移时轻则报错、重则静默错配（UI 以为支持某能力，server 实际没有）；`not_initialized` 是刻意的门禁（见 §7）。                   |
| `**transport` 适配**     | 把不同传输介质的原始 I/O 统一译成内部消息模型：`stdio` 按行 NDJSON、`WebSocket` text frame、未来 `ACP` JSON-RPC 等，均经 adapter 进出同一套 `ServeCommand` / `OutFrame` dispatcher。               | 不管走管道还是网线，先翻译成同一种内部消息，再走同一套命令/事件/控制逻辑。 | **协议/业务与传输解耦**（对标 codex in-process + stdio/WS 共用 dispatcher、hermes 同一 handler 表走 stdio/WS）：Phase 1 上 stdio，Phase 2 加 WS/ACP 不重写 `commands`/`event_pump`/`control`。 | 不做适配层则业务代码被 `stdin`/socket/JSON-RPC 细节污染；每加一种传输就复制一套逻辑，最终分叉失控（openclaw 纯网关 + embed 重复实现的反面教材）。 |
| **单写者 `writer drain`** | 所有下行帧（命令响应、事件、控制请求）先入唯一 `mpsc` 队列，由**唯一 writer 任务**顺序 `drain`：序列化（`ndjson_safe_stringify`）→ `write_all` + `flush`；可叠加 per-session 背压、delta 合并、跨 session 公平轮转。 | 谁都别直接往 stdout/socket 抢写，先排队，由一个出口统一发。  | 下行来源多且跨线程并发（`event_pump`、命令回包、`control`）；多 session 时多路事件同时入队。单出口保证**帧完整、顺序可预期**；再按 `sessionId` demux 给 UI，避免热会话饿死其它 tab（见 §3.3 / R9）。                              | 多写者必然 NDJSON 行交错、半行损坏，UI 解析必崩（cc-fork 单 FIFO drain 的教训）；无 drain 则无法在出口统一做背压与 lossless 生命周期帧保护。 |


读图导读（说人话）：这张图现在故意把「**连接级**」和「**会话级**」拆开画。UI 虽然只连到一个 `tomcat serve` / gateway 进程，但同一条连接上可以同时挂多个 `sessionId`：Tab-A 发 `s1`，Tab-B 发 `s2`。中间那层 dispatcher 不再只是“拥有一次 run”，而是持有一张 `ChatContextRegistry`，里面并排放着多个会话槽；每个槽各自一条 run 生命周期、各自一份中断信号，所以 **跨 session 能真并发，同 session 仍串行**。三类流并没有变：①上行「命令」、②下行「事件」、③双向「控制」；只是现在它们都显式带上 `sessionId`（审批再带 `request_id`），于是命令能按会话路由，事件能按会话 demux，控制能准确回到正确 tab。核心仍是现有的 `AgentLoop` + `EventBus`，唯一的外沿分叉仍然只是传输：先 stdio，WS/HTTP daemon 是后话。

读图导读（说人话）：这张图把 A.1 的抽象层落到真实文件。关键点有五个：（1）入口是 `src/api/cli/mod.rs` 里**新增一个与 `Claw`/`Code` 平级的 `Serve` 子命令**，不动其它命令；（2）`src/api/serve/` 是新模块，但它**复用** `ChatContext::from_config`、`api/chat/run_loop`，不重写 agent；（3）**多会话并发**靠新增 `registry.rs` 的 `ChatContextRegistry`——这正是 `multi-agent.md` 维度A/MA2 规划的那张表，serve 进程里 `sessionId → (ChatContext + 一个 AgentLoop run task + cancel_token)`，进程级 `Arc<dyn LlmProvider/PrimitiveExecutor/EventBus>` 共享、会话上下文隔离；（4）下行事件**完全复用**现有 `ScopedEventEmitter::emit → WireEnvelope → EventBus.emit_sync` 链路（事件类型一个都不用新增），`payload.sessionId` 已经天然带上，`writer` 直接拿它把 s1/s2 多路事件 demux 回各自客户端；（5）审批/提问复用既有 `ask_question_wire.rs` 的 EventBus request/response 回环，控制帧用 `request_id + sessionId` 双重路由（codex 教训）。Phase 2 的 WS 网关只是把 `writer/stdin` 换成 axum WebSocket 适配器，**dispatcher 与 registry 不变**。

### B. 状态机（一条连接 / 一次 turn 的生命周期）

```text
                       initialize(client_info)            prompt{id,text}
   ┌────────────┐  ───────────────────────────►  ┌──────────┐  ─────────────►  ┌───────────┐
   │ connected  │                                 │  ready   │                  │ running   │
   │ (待握手)   │  ◄── initialize_result(caps) ── │ (空闲)   │                  │ (turn 中) │
   └────────────┘                                 └────┬─────┘                  └─────┬─────┘
                                                       ▲                              │
                              agent_end / agent_interrupted                           │ control_request
                                                       │                              │ (ask_question/审批)
                                                       │                              ▼
                                                       │                        ┌───────────────┐
                                                       │  control_response      │ awaiting_user │
                                                       └────────────────────────┤ (阻塞等回包)  │
                                                                                 └───────┬───────┘
                                                       interrupt                         │ 回包到达
                                                  ┌──────────────┐  ◄───────────────────┘
                            running ── interrupt ─►│ interrupting │── 收口 ─► ready
                                                  └──────────────┘
```


| 当前状态                    | 事件                               | 目标状态          | 副作用                                        | 说人话                    |
| ----------------------- | -------------------------------- | ------------- | ------------------------------------------ | ---------------------- |
| connected               | `initialize`                     | ready         | 回 `initialize_result`（能力位、协议版本、sessionId）  | 先握手，告诉 UI「我支持哪些命令/事件」。 |
| ready                   | `prompt` / `steer` / `follow_up` | running       | 驱动 `AgentLoop.run()`；发 `agent_start`       | 用户发话，agent 跑起来。        |
| running                 | `tool/message/usage` 事件          | running       | 经 writer 单写者 drain 到下行                     | 流式可观测，不改状态。            |
| running                 | `plan.ask_question` / 审批门禁       | awaiting_user | 发 `control_request{request_id}`，**阻塞工具继续** | 必须等用户拍板才能往下。           |
| awaiting_user           | `control_response{request_id}`   | running       | emit 回 EventBus 的 response_event，解阻塞       | 用户答了，接着干。              |
| running / awaiting_user | `interrupt`                      | interrupting  | 置该会话中断信号；已发 Start 的工具必补 End            | 软中断也要把 UI 收口。          |
| interrupting            | run 收敛                           | ready         | 发 `agent_interrupted` 或 `agent_end{error}` | 回到空闲，等下一条。             |
| running                 | run 自然收敛                         | ready         | 发 `turn_end` → `agent_end`                 | 正常结束一回合。               |


**多会话视角（本期目标）**：上面这张状态机描述的是**单个会话（sessionId）的生命周期。一条连接可同时承载多个会话，因此进程内每个 `sessionId` 各自跑一份独立状态机**（各自 `ready/running/awaiting_user/interrupting`），互不阻塞：s1 在 `awaiting_user` 等审批时，s2 仍可在 `running` 流式。`ChatContextRegistry` 持有这些会话槽，`writer` 用 `WireEnvelope.sessionId` 把多路状态流 demux 回客户端。并发粒度对齐 codex：**跨会话真并发、同一会话内串行**（一个 `sessionId` 同时只有一个活跃 turn，再来 `prompt` 走 `steer/follow_up` 入队或返回 `busy`，对齐 hermes `4009 session busy` 语义）。`initialize` 握手是**连接级**一次性动作，不随会话重复。
