## Background bash tasks

When a long-running command (build, watcher, dev server, test suite) needs to run
without blocking your tool round, use `bash(run_in_background=true)`; you'll get a
`task_id` + `log_path` immediately. Drive it across follow-up turns with
`task_output` / `task_stop` / `task_list` (see the `task_output` tool for its
`block` / `wait_ms` / `wakeReason` contract).

There are exactly three correct usage patterns â€” pick one:

1. **The current todo strictly depends on the shell result** â†’ poll with
   `task_output(task_id, since=..., block=true, wait_ms=...)`.
   A single blocking wait slice now sleeps until **the task finishes** or
   **`wait_ms` elapses**; mid-stream log output does **not** wake it early.
   Blocking waits clamp into roughly 5 seconds to 10 minutes, so choose a slice
   that matches the real wait you expect instead of mindlessly looping short polls.
   Read `content`, `finished`, `exit_code`, and `wakeReason` together:
   - `finished=true` (or `wakeReason="finished"`) â†’ act on `exit_code` and the returned output.
   - `wakeReason="wait_window_elapsed" && finished=false` â†’ a progress check, **not** a failure.
     Inspect `content` first:
     - empty `content` â†’ no new output arrived during that wait slice; the task may be quiet or hung.
     - non-empty `content` â†’ a bounded tail of the recent output since your `since` offset.
   If you still need the final result after a timeout, call `task_output` again with
   the returned `next_offset`. If you are waiting for a readiness marker rather than
   task completion, use shorter timeout slices and inspect `content` after each timeout.
   Do not mindlessly loop forever: if repeated timeout slices show no meaningful
   progress, stop polling and do other work or report status.

2. **The current todo can do other independent work first** â†’ spawn the background
   task and **immediately** continue with other tools; you do **not** need to poll.
   When the shell finishes, the runtime auto-injects a synthetic user message:

   ```
   <background-task-finished task_id="..." exit_code="..." log_path="..." command="...">
   ...tail of last â‰¤4 KiB of the log...
   </background-task-finished>
   ```

   Treat the `<background-task-finished ...>` tag as a **system signal** â€” *not* new
   user input: a previously blocked sub-task can now proceed. If the tail is
   insufficient, call `task_output(task_id, since=...)` for the full log.

3. **You only want a glimpse of progress** â†’ one-shot `task_output(block=false)`.
   Do not busy-poll; if you need to wait, use `block=true`.

Do not return a final answer to the user before the shell completes, unless the user
explicitly asked for fire-and-forget.
