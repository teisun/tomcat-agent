## Background bash tasks

When a long-running command (build, watcher, dev server, test suite) needs to run
without blocking your tool round, use `bash(run_in_background=true)`; you'll get a
`task_id` + `log_path` immediately. Drive it across follow-up turns with
`task_output` / `task_stop` / `task_list` (see the `task_output` tool for its
`block` / `timeout_ms` / `wakeReason` contract).

There are exactly three correct usage patterns — pick one:

1. **The current todo strictly depends on the shell result** → poll with
   `task_output(task_id, since=..., block=true, timeout_ms=...)`.
   A single blocking wait slice can now cover roughly 5 seconds to 10 minutes,
   so prefer a longer slice when you truly need to wait instead of mindlessly
   looping short polls.
   Read `content`, `finished`, `exit_code`, and `wakeReason` together:
   - `finished=true` → act on `exit_code` and the returned output.
   - `wakeReason="new_output"` → use the returned bytes to decide the next step.
   - `wakeReason="timeout" && finished=false` → a progress check, **not** a failure;
     inspect the returned output/tail first, only wait again if you truly must.
   Do not mindlessly loop forever: if repeated timeout slices show no meaningful
   progress, stop polling and do other work or report status.

2. **The current todo can do other independent work first** → spawn the background
   task and **immediately** continue with other tools; you do **not** need to poll.
   When the shell finishes, the runtime auto-injects a synthetic user message:

   ```
   <background-task-finished task_id="..." exit_code="..." log_path="..." command="...">
   ...tail of last ≤4 KiB of the log...
   </background-task-finished>
   ```

   Treat the `<background-task-finished ...>` tag as a **system signal** — *not* new
   user input: a previously blocked sub-task can now proceed. If the tail is
   insufficient, call `task_output(task_id, since=...)` for the full log.

3. **You only want a glimpse of progress** → one-shot `task_output(block=false)`.
   Do not busy-poll; if you need to wait, use `block=true`.

Do not return a final answer to the user before the shell completes, unless the user
explicitly asked for fire-and-forget.
