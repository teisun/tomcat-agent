# Architecture Spec 规范

> 适用：`docs/architecture/**/*.md`（跨模块技术方案）。
> 标杆案例：[`architecture/tools/search_files.md`](../../../../docs/architecture/tools/search_files.md)、[`architecture/tools/read.md`](../../../../docs/architecture/tools/read.md)（**§7.1 / §7.2 写法**）、[`architecture/tools/edit.md`](../../../../docs/architecture/tools/edit.md)（原则 / 选型 / 协议 / 状态机 / 测试等**多表带「说人话」列**范例）、[`architecture/interrupt-and-cancellation.md`](../../../../docs/architecture/interrupt-and-cancellation.md)（**§7.0 豁免**：回链主方案）。
> 上位规范：[`Constitution.md`](../../Constitution.md)（**二.10「说人话」辅助**）· [`DOCUMENTATION_GUIDE.md`](DOCUMENTATION_GUIDE.md)。

## 1. 何时写一份新方案

任一即触发：

1. 设计跨 ≥ 2 个 `src/` 一级子目录的行为；
2. 新增 / 修改 wire 协议 / 事件 / 错误契约；
3. 引入新的状态机 / 生命周期 / 取消传播；
4. 计划文档（`agents/plan/*`）显式写"开发前需新增架构文档"（参见 [`PLAN_SPEC.md §2`](../../../../agents/plan/PLAN_SPEC.md)）。

模块 README 描述"本模块做什么"；架构方案描述"一件事跨多个模块怎么串起来做"。两者并存、不互相替代。

## 2. 章节骨架（推荐顺序）

| 章节 | 硬度 | 何时必出 |
|------|------|----------|
| §1 目标与设计原则 | **MUST** | 所有方案 |
| §2 术语统一 | **MUST** | 任何易混淆的命名（`interrupt / cancel / abort` 之类） |
| §7 选型与竞品对比（**至少**含 §7.1 **落地选型决策表** + §7.2 **实施点（已闭环）**） | **MUST** | 所有方案；**§7.0 豁免**见下 |
| §3 协议（入参 / 出参 / Schema） | **MUST**（涉及对外接口时） | 引入或修改工具 / 事件 / Trait / wire 时 |
| §4 文件职责总览（One-Glance Map） | **MUST** | 所有方案 |
| §5 调度时序（运行时图） | **SHOULD** | 跨模块数据流 / 多线程 / 异步 / 取消传播 |
| §6 状态机 | **SHOULD** | 有显式生命周期 / 状态迁移 |
| §8 配置与环境变量 | **SHOULD** | 引入新配置项 / env 开关 |
| §9 错误模型 / 截断 / 警告 | **SHOULD** | 有非 panic 的归一化结局（warning / truncated 等） |
| §10 测试矩阵（验收） | **MUST** | 所有方案 |
| §11 风险与应对 | **MUST** | 所有方案 |
| §12 历史决策 / 跨文档修订 | **SHOULD** | 有被取代的旧方案 / 影响相邻方案文档 |

> **行序**：**§7** 紧挨 **§2**，表示「术语钉死用词 → 选型与实施钉死弃案与交付 → 再下钻协议与 One-Glance」。具体 `*.md` 里可用 **§2** 承接整章竞品/选型（语义即本规范的 **§7**），不必强行改文档内章节号为 7。新稿若易混淆术语（如 `overwrite` / `stamp`），**宜**采用 **`write.md` 式**：**§2 术语统一 → §3 选型与实施**（与上句精神一致，仅文档内编号提前术语章）。
>
> 章节编号可酌情合并（如 §5 + §6 合成"调度与状态机"），但**MUST 项一个不能省**。

## 3. 各节硬约束

### §1 目标与设计原则（MUST）

- 用**观察指标表**替代散文："2 秒内子进程死亡"、"truncated=true 时必给 next_offset"。
- 显式列**非目标**及"推给谁 / 哪个任务号"，防止范围蔓延。

```markdown
| 目标 | 观察指标（落地后用户可感知） | 说人话 |
|------|------------------------------|--------|
| G1   | Ctrl+C 后 2 s 内 child 进程死亡 | 用户按停之后，子进程别拖着不死。 |

| 非目标 | 推给 | 说人话 |
|--------|------|--------|
| 跨 session 续跑 | T2-P1-001 | 这条本期不做，交给看板那条任务。 |
```

「说人话」表格列的写法见 **§4.1** 下「表格末列」；§1 观察指标表与非目标表 **SHOULD** 带该列。

### §2 术语统一（MUST）

每条术语三件套：

- **语义**：一句大白话。
- **数据载体**：`#[serde(skip)]` 字段 / `Arc<Mutex<...>>` / 某个 wire 常量等。
- **行为约束**：触发 / 失效 / 唯一性 / 与相邻术语的互斥关系。

若术语以**表格**呈现：在保留三件套的前提下，**SHOULD** 在表末追加 **`说人话`** 列（更短口语，见 **§4.1**）；与「语义」列重复度过高时可合并为一列，但须保留一句纯口语。

模糊时间词（"LLM 回复后"、"发起下一次请求前"）必须在首次出现处用一句话钉死指代位置。

### §3 协议（涉对外接口时 MUST）

用**字段表 + 子结构 + 调用样例**三件套描述：

| 字段 | JSON 类型 | 必填 | 默认值 | 适用场景 | 说明 | 说人话 |
|------|-----------|------|--------|----------|------|--------|
| ... | ... | ... | ... | ... | ... | ... |

要求：

1. 列出每一个字段（类型、必填、默认值、约束）；可选字段要说三态语义（缺省 / 显式 null / 显式值）。
2. 字段表 **SHOULD** 在末列加「说人话」（见 **§4.1**），便于非实现者扫读。
3. 子结构 / 嵌套 enum 单独画 ASCII 框图或代码块。
4. 至少一组**输入 + 输出**的调用样例（jsonc 注释形式可读性最佳）。
5. 标明**单一事实源**（哪一个 `*.rs` 是协议的最终定义）。

### §4 文件职责总览（One-Glance Map，MUST）

ASCII 框图，**不可省**。硬约束：

1. **覆盖完整**：方案改到的每个 `*.rs` 都有节点，缺一不可。
2. **箭头串联**：节点之间用 `│ + ▼` 或 `→` 标方向；自顶向下读一遍即复现整条链路。
3. **节点内要点**：列函数 / 类型 / 常量 / 关键行为，不允许只写文件名。
4. **挂独立 tests.rs**：节点内或下方注明配套 `[*/tests.rs]`（按 [`RUST_FILE_LINES_SPEC §A`](../coding/RUST_FILE_LINES_SPEC.md)）。
5. **图后阅读顺序**：紧跟 2–3 句 **「说人话」**（口语串链路），标清自顶向下谁调谁，与 **§4.1** 段落级要求一致。
6. **实施期偏离要标注**：如签名最终未改，节点上加 `【未改签名 / 依赖 Drop】` 之类显式标签，避免 stale 描述误导。

反例：

- ❌ 只画分层框（infra → core → api），不落到文件级。
- ❌ 节点只有文件名，没有函数 / 行为。
- ❌ 涉及 5 个文件，图里只画 3 个。

### §5 调度时序（SHOULD）

按需组合：

- **多列 ASCII 时序图**：如 [`search_files.md §5`](../../../../docs/architecture/tools/search_files.md)，按"用户 / 入口 / 调度层 / 子进程"排列泳道；终端友好、易 diff。
- **Mermaid `sequenceDiagram autonumber`**：用于在 IDE 里渲染。
- **控制流伪码**：复杂嵌套循环 / 状态迁移用文字伪代码（如 [`agent-loop.md §13.3.2`](../../../../docs/architecture/agent-loop.md) 三层循环）。

每条事件 / 状态迁移必须显式标"何时发布、订阅方在哪"。

### §6 状态机（SHOULD）

用 ASCII 状态图 + 转移表二选一或并用：

```text
┌─────────┐ event_a  ┌─────────┐ event_b  ┌─────────┐
│  init   │─────────▶│ running │─────────▶│  done   │
└─────────┘          └────┬────┘          └─────────┘
                          │ cancel
                          ▼
                    ┌──────────┐
                    │  failed  │
                    └──────────┘
```

| 当前状态 | 事件 | 目标状态 | 副作用 | 说人话 |
|----------|------|----------|--------|--------|
| init | event_a | running | emit Started | 收到启动事件就进入跑起来的状态。 |
| running | cancel | failed | emit Failed{reason}, kill child | 用户取消就失败收尾并把子进程干掉。 |

### §7 选型与竞品对比（MUST）

把「读过竞品 / 备选路线后的结论」与「怎么分批合进主线」钉在 spec 里，避免只在 `agents/plan/*` 口头对齐、正文只剩首屏散文。**竞品横向表 / 形态分类图**仍按主题 **SHOULD** 展开（无成熟对标时可缩短为一段「无可比公开协议」声明 + 自研约束）。

#### §7.0 章节编排与豁免

- **推荐**：与 [`search_files.md`](../../../../docs/architecture/tools/search_files.md)、[`read.md`](../../../../docs/architecture/tools/read.md) 一致，把「关切 / 横向对比 + **§7.1** + **§7.2**」放在**协议与 One-Glance 之前**，便于 reviewer 先吃决策再下钻契约（章节编号可为文档内 **§2**，语义上仍对应本规范的 §7）。
- **豁免（不必重复长竞品表 + 长实施表）**：文首声明为**另一份架构主文档的补充**（例：[`interrupt-and-cancellation.md`](../../../../docs/architecture/interrupt-and-cancellation.md) 对 [`agent-loop.md`](../../../../docs/architecture/agent-loop.md)），且本文**不引入**新的对外工具名 / wire 字段时：**须**在正文**显式回链**主文档。**§7.1 / §7.2** 二选一落地：（**a**）被链主文档（或其已链子文档）中**已有**与本交付对齐的等价决策表 + 实施点表 → 本文不重复；（**b**）否则本文须在「选型 / 原理 / 背景」任一章内给出**至少各一行**的精简 **§7.1** + **§7.2** 表（含 PR 或看板锚点 + **验收函数名**），竞品横向可省略。若本文提出**新的**弃案或新的分阶段交付，则必须在本文内**增补对应表格行**。

#### §7.1 落地选型决策表（MUST）

在选型章内给出**至少一张**决策矩阵，**二选一**形态（信息密度须等价，禁止只写一句「我们选了 X」）：

**形态 A（推荐，前四列列名一字不差）**：

```markdown
| 决策点 | 默认选择 | 主要替代方案 | 选择理由（为何不是替代） | 说人话 |
|--------|----------|--------------|--------------------------|--------|
```

前四列语义与审计要求不变；末列「说人话」为 **SHOULD**（见 **§4.1**）。未加该列不视为缺 §7.1，但**新增或大面积改版**的架构文档宜补上。

**形态 B（路线对比表，与 [`search_files.md` §2.3](../../../../docs/architecture/tools/search_files.md) 同类）**：多列对比若干候选路线（如 A/B/C），表下**必须**用一段话或短列表钉死 **最终默认 = 哪一列 / 哪一组合**，并写明 **弃用列及原因**（等价于形态 A 的「为何不是替代」）。**SHOULD** 在表末追加「说人话」列，每行一句扫读用语。

硬约束：

1. **一格一事 / 一行一路线**：每个可辩驳分叉独占表的一行（或独占路线列），禁止把整章结论揉成一行笼统话。
2. **「为何不是替代」可审计**：读者能回答「若不选默认，会付出什么代价」（形态 B 可由表下「最终决策」段承担）。
3. **确无并列弃案**的极少数单一路线：保留表头或等价标题，用一行写清 **「实行单一路线」** / 替代方案 **无** / 理由链 PR 或主方案 —— **禁止**删节整个决策矩阵。

#### §7.2 实施点（已闭环）（MUST）

与 §7.1 **同章相邻**（前或后均可），给出实施点总表（**前四列**列名与语义**一字不差**；末列「说人话」按 **§4.1** 为可选扩展）：

```markdown
| 实施点 | 交付范围 | 主要代码落点 | 验收锚点（示例） | 说人话 |
|--------|----------|--------------|------------------|--------|
```

末列「说人话」为 **SHOULD**（见 **§4.1**）。前四列要求不变。

硬约束：

1. **SHOULD（强烈建议）**：表后按实施点拆小节（如 `read.md` **§2.4.1** 起），每小结配 **技术要点** + **ASCII**（或显式写「见图：本文 §5 / §6」），帮助把「表上一格」映射到调用栈与数据流。完整范例：[`read.md` §2.3–§2.4](../../../../docs/architecture/tools/read.md)；**同构另一例**（术语章在前、实施点为 **§3.4**）：[`write.md` §3.4.1](../../../../docs/architecture/tools/write.md)。
2. **单行压缩（仅当单次合入、无子 PR 叙事）**：允许全表 **仅一行**——`实施点` 写 `单次合入` 或主 merge / tag；`验收锚点` 列**必须**含「见本文 **§10 测试矩阵**（§ 号随文档）」**或**直接列出 **≥1** 个测试函数名；**禁止**写「已测」而无指针。
3. **与 1 的边界（避免「含糊」）**：本条 **SHOULD** 的强度随**实施点行数**变化——表内 **≥2 行**（或多 PR、多交付边界）却**无**对应拆节、又无等价「见图 §x / §y」指针时，属可读性 **SHOULD** 未落实，review **可要求**作者补 **§7.2.1** 式小节；**不**自动等同 MUST 缺项。仅 **一行**且满足上条 **「单行压缩」** 时，**可不拆**子节。

竞品横向对比模板（§7 正文 **SHOULD** 保留或等价浓缩）：

```markdown
| 竞品 | 形态 | 关键设计 | 我们借鉴的点 | 说人话 |
|------|------|----------|---------------|--------|
| ripgrep | spawn | 默认遵守 .gitignore | Tier1 内容主力 | 默认就尊重 gitignore，跟 rg 习惯一致。 |
| ignore crate | 进程内 | WalkBuilder + filter_entry | Tier2 兜底主力 | 进程内遍历时用成熟库做过滤。 |
```

在有多条候选路线时，仍须给出「为什么选 A 不选 B」的 **3–5 条**理由（可写在决策表或紧随其后的短列表）。

### §8 配置与环境变量（SHOULD）

```markdown
| 变量 | 取值 | 含义 | 优先级 | 说人话 |
|------|------|------|--------|--------|
| `PI_SKIP_X` | `1` / `true` | 关闭 X | env（最高） | 环境变量一设就关掉某功能。 |
| `[section] x_enabled` | bool | 配置文件 | config | 配置文件里也能开关。 |
```

总则一律写明：**env > config > 默认**。

### §9 错误模型 / 截断 / 警告（SHOULD）

用 ASCII 框图列出**所有归一化结局**：哪些走 `Err`、哪些走 `truncated=true + warnings`、哪些静默兜底。避免读者脑补错误处理路径。

```text
正常 → warnings += "implementation=tier1"
分页截断 → truncated=true + next_offset
墙钟到点 → truncated=true + warning
容忍错误 → 空命中 + warning（不抛 Err）
致命错误 → Err（参数校验 / 权限拒绝 / IO 致命）
```

### §10 测试矩阵（MUST）

映射到**实际测试函数名**：

| 维度 | 用例 / 编号 | 状态 | 说人话 |
|------|-------------|------|--------|
| 单元 | `mod::tests::name`（n 用例） | ✅ 日期 | 模块内小测锁住细节。 |
| 集成 | `tests/foo_tests.rs::case_name` | ✅ | 黑盒或跨模块拼起来测。 |
| E2E | `E2E-XXX-NNN`（[`E2E_SCENARIO_LIBRARY.md`](../testing/E2E_SCENARIO_LIBRARY.md)） | PENDING | 用户能摸到的整条链路。 |
| 观察指标 | G1–G5 对应的锁死机制 | ✅ | §1 里吹的牛，这里用测试钉死。 |
| 文档 | 本文定稿 + 相邻方案同步 | ✅ | 字和代码别两张皮。 |

要求：

- §1 里每条目标在本表中能找到"锁死它的测试 / 机制"。
- 状态列只允许 `✅ 日期` / `PENDING` / `阻塞于 X`，禁止留空或写"待补"。

### §11 风险与应对（MUST）

```markdown
| 风险 | 影响 | 应对（具体动作） | 说人话 |
|------|------|--------------------|--------|
| 大仓库 Tier2 超时 | 中 | 10 s 墙钟 + truncated=true + warning；env 可调 | 慢就截断返回，别无限等。 |
```

应对必须落到具体动作（"加 `tokio::time::timeout(2s)`"、"`fsync` 后再返回"）。至少覆盖：并发 / 信号 / 兼容性 / 回滚 / schema 破坏 / panic 隔离 / 资源泄漏 中相关项。

### §12 历史决策 / 跨文档修订（SHOULD）

- 用删除线 + "→ 否：理由"形式列出**已被取代**的方案，让 reviewer 一眼看到设计是怎么演化到当前的。
- 若本方案改写了相邻方案文档某段语义，单开"跨文档修订"小节登记修订意图（即使本 PR 暂未同步改）。

## 4. 写作风格（与 [`MODULE_README_SPEC.md`](MODULE_README_SPEC.md) 共用）

| 项 | 要求 |
|----|------|
| 精准引用 | 文件 / 符号 / 行号都尽量带路径 |
| 术语一致 | 与术语表保持，不允许同义词混用 |
| 时间点钉死 | "LLM 回复后" 一类模糊词必须用一句话约定指代位置 |
| Why 优先 | 先说"为什么"，再说"怎么做"，再说"做了什么" |
| ASCII 优先 | 终端友好、易 diff；复杂时序可补 mermaid |
| 表格优先 | 选型 / 状态转移 / 风险 / 验收 / 配置等结构化内容用表格 |
| **说人话** | 见 **§4.1**（**先专业、后口语**：段落辅助 + 表格末列）；与 [`Constitution.md`](../../Constitution.md) **二.10** 同向 |
| No-Stale | 实施期调整须就地更新文档，并标 `【未改签名 / 依赖 Drop】` 等显式标签；禁止留"原计划/现落地"两份真相 |
| 短句收尾 | 篇幅 > 300 行的方案末尾补一段"一句话总结"（参考 [`interrupt-and-cancellation.md §4.6`](../../../../docs/architecture/interrupt-and-cancellation.md) 末尾） |

### 4.1 「说人话」辅助理解（SHOULD）

落实 [`Constitution.md`](../../Constitution.md) **二.10**。**总顺序**：**先**写清、写准专业技术内容（定义、契约、步骤、边界、可归一化结局）；**再**用口语化文字帮助读者扫读。**禁止**用「说人话」替代必须的技术表述。

#### 段落或小节末尾（含图后「阅读顺序」）

- 在每个**主要**技术小节、或整段 ASCII / Mermaid 技术描述之后，**SHOULD** 可紧跟 **2–5 句**口语串意图；可用连续段落，或小标题 **`说人话`** / **`阅读顺序（说人话）`**。
- **One-Glance Map** 图后的「阅读顺序」段归入本类（与 **§3「文件职责总览」** 硬约束 5 一致）。
- **豁免**：纯目录、纯索引、过渡句已足够短、或该块正文本身已是低密度说明时，不强制再加。

#### 表格末列「说人话」

在信息密度高的 Markdown 表末追加一列，列名固定为 **`说人话`**，与 [`architecture/tools/edit.md`](../../../../docs/architecture/tools/edit.md) 等标杆一致。

**何时 SHOULD 加**

- 原则 / 观察指标（G 表）、非目标、竞品横向、现状基线、维度→落地点矩阵、**§7.1** 决策表、**§7.2** 实施点总表、按 `#` 编号的实施落地点小节表、协议字段表、状态机转移表、配置表、错误归一化表、测试矩阵、风险表、历史决策表、附录「旧节号 ↔ 新版」对照表等。

**写什么**

- 每行 **一句**口语化中文（可含少量术语），供扫读与 onboarding。
- **不**复述同行技术列的完整定义，**不**替代「观察指标 / 字段说明 / 为何不是替代」正文列；与 **§2 术语表「语义（大白话）」** 可并存：术语表以三件套为主；若表内已有「大白话」列，「说人话」宜更短，作页内速记。

**豁免（可省略该列）**

- 纯路径 / 符号索引、或列数 ≤2 且每格已是一句完整大白话的极短表。
- **§7.0 豁免**的补充文档：若主文档已含完整「说人话」列且本文不新增可辩驳分叉，可不重复；若本文新增决策行，**SHOULD** 为新增行补「说人话」。

**与 MUST 的关系**

- 缺「说人话」列或缺段落辅助 **不**构成缺 §7.1 / §7.2 / §10 等 MUST 项；属可读性 **SHOULD**。新增或大面积改版架构文档时优先补齐。

## 6. 反例汇总

- ❌ §1 用散文："本方案旨在改善搜索体验"——必须翻成观察指标表。
- ❌ 协议节漏掉默认值或必填标记。
- ❌ One-Glance Map 只画分层，不落到文件级。
- ❌ 状态机用一段散文描述："收到事件后会进入 running 状态" —— 必须配 ASCII 状态图 + 转移表。
- ❌ 风险节写"注意一下并发问题"——必须到"加 `tokio::time::timeout(2s)`"这种动作粒度。
- ❌ 验收节"已通过测试"——必须列测试函数名 + 状态。
- ❌ 选型章只有散文或只列「我们用了 X」——缺 **§7.1 落地选型决策表** 或 **§7.2 实施点（已闭环）**（且无 §7.0 豁免回链）。
