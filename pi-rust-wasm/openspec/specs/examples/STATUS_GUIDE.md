# 进度状态文件规范 (Status Tracking Spec)

## 1. 核心规则
- **命名规范**：文件名必须与当前 Git 分支一一对应。
  - 转换规则：`分支名 / 替换为 -`，并存放于 `status/` 目录。
  - 若非 `feature/*` 分支, 则直接用分支名
  - 示例：`feature/infra-bus` -> `status/feature-infra-bus.md`, `develop` -> `develop.md`。
- **排版约束**：
  - **禁止** 使用 `#` (H1) 和 `##` (H2) 标题。
  - 汇总脚本会自动以分支名作为 H2，本文件内仅允许使用 `###` (H3)。
- **更新频率**：每次 `git commit` 前或每日结束前更新。

---

## 2. 标准模板 (Template)

```markdown
| Owner | Update Time | State | Branch |
| :--- | :--- | :--- | :--- |
| @YourName | YYYY-MM-DD HH:mm | [ACTIVE/PAUSED/DONE] | branch-name |

### ✅ DONE (已完成/进行中)
- [✓] **[P0]** 基础设施层核心 Trait 定义 @date
- [ ] **[P1]** 事件总线异步分发优化 (进行中 70%)
- [ ] **[P2]** 单元测试覆盖率提升至 90%

### 🔌 INTERFACE (接口变更)
> 记录本分支引入的破坏性变更或新增对外 API，方便其他成员对接。
- `EventBus::emit_async`: 新增 `priority` 参数。
- `AppConfig`: 增加 `security.audit_log` 配置项。

### ⚠️ BLOCKED (阻塞/风险)
| 阻塞项 | 原因 | 预计解决 |
| :--- | :--- | :--- |
| WASM 运行时 | WasmEdge 1.0 接口不兼容 | 2024-xx-xx |
| 无 | - | - |
```

---

## 3. 规范细节说明

### 3.1 元数据 (Metadata Table)
- **State 取值**：
  - `ACTIVE`: 正在积极开发中。
  - `PAUSED`: 暂时挂起（需在 BLOCKED 中说明原因）。
  - `DONE`: 本分支任务全部完成，准备合并。
- **Update Time**: 必须精确到分钟，用于看板识别最新的提交状态。

### 3.2 任务列表 (Tasks)
- **优先级标记**：建议在任务前增加 `[P0]` (最高)、`[P1]`、`[P2]`。
- **状态标记**：
  - `[✓]` 表示已完成且通过测试。
  - `[ ]` 表示进行中或待办。
- **百分比**：对于大任务，建议标注进度，如 `(50%)`。

### 3.3 接口变更 (Interface)
- 如果本分支不涉及接口变动，请写“无显著变更”。
- 涉及变更时，需写出具体的 **函数名**、**结构体名** 或 **配置路径**。

### 3.4 阻塞与风险 (Blocked)
- 严禁空着不写。若无阻塞，请填“无”。
- 协作场景下，这里是请求 Leader 或同事介入的最快入口。

---

## 4. Cursor / AI 指令 (Prompt)

当你让 Cursor 更新进度时，可以输入：

> **Prompt:** "请根据我刚才的代码修改，更新 `status/` 下对应的进度文件。要求：
> 1. 更新元数据表格中的时间为当前。
> 2. 在 DONE 区域勾选已完成的任务，并根据代码新增实际产出的任务项。
> 3. 如果我修改了任何公开的 Trait 或 Struct，请在 INTERFACE 区域列出变更。
> 4. 保持 H3 标题格式，严禁使用 H1/H2。"

---

## 5. 优化点总结 (对比原版)
1. **增加了 P0/P1 优先级概念**：看板一眼就能看出核心任务是否受阻。
2. **强制表格对齐**：使用 `:---` 确保在任何 Markdown 预览器中都整齐。
3. **接口部分结构化**：不再是纯描述，而是强调“变更点”，更具技术价值。
4. **阻塞部分表格化**：清晰展示“原因”和“解法”，便于项目管理。
