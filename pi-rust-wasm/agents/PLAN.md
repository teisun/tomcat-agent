# 001-mvp 多 Agent 分工总计划

## 并行度建议

- **首轮**：仅 1 个开发 Agent 完成 T1-P0-001（项目骨架与基础设施），其余等待。
- **001 完成后**：最多 **4 个开发 Agent 同时开工**，对应 T1-P0-002、003、004、007 四条线。
- **后续波次**：005/006（2 条）→ 008（1 条）→ 009（1 条）→ 010/011（2 条），按依赖顺序推进。
- **测试集成**：1 名 **integration_test** 不占开发任务，负责合并分支到 develop、全量测试与验收。

**推荐配置**：**4 开发 + 1 集成** 同时在线（在 002/003/004/007 阶段）；其余阶段按依赖表依次启动或收尾。

每一波次的「任务 ID」仅表示该波次的**主任务编号**。真实需要做的**完整范围**见下文的「波次任务范围说明」逐条；子项以 [tasks_details.md](../openspec/changes/001-mvp/tasks_details.md) 为准。

- **波次 3** 主任务 ID：T1-P0-005、006。依赖：001、002。负责角色：primitives_tools（002 由 infra 交付）。
- **波次 4** 主任务 ID：T1-P0-008。依赖：005、006、007。负责角色：wasm_plugin（**接续 007**，即本波次 = 007 未完成部分 + 008 全量）。

### 阅读与执行说明（防遗漏）

- **任务 ID 仅为主任务**。完整范围以本页「波次任务范围说明」的逐条描述为准；各任务子项以 [tasks_details.md](../openspec/changes/001-mvp/tasks_details.md) 为准（例如 001 对应 1.1–1.8，007 对应 7.1–7.6、7.x）。
- **「接续」含义**：标注了「接续 XXX」的波次 = 本波次除完成主任务外，**必须同时完成前一任务的未完成部分或后续落地**。例如：第 4 波 wasm_plugin 表示 007 接续 + 008 全量；第 6 波 session_cli 表示 003 接续（若有）+ 010 全量。**常见遗漏**：只做「任务 ID」对应内容而忽略接续，导致前一任务长期为桩或半成品。
- **单波次计划必须做到三件事**：① 先查本页「波次任务范围说明」中该波次、该角色的那条，确定完整范围（含接续）；② 制定实施计划时**同时写出接续与主任务两部分的实施步骤**（例如 007 接续实施步骤 + 008 实施步骤）；③ 对照 tasks_details 逐子项核对，不得只写主任务 ID 的步骤。

## 分支与集成策略

- **分支约定**
  - 主开发分支：`develop`
  - 功能分支：`feature/infra`、`feature/session-cli`、`feature/llm`、`feature/wasm-plugin`、`feature/primitives-tools`、`feature/chat`（与角色一一对应）
- **合并顺序**
  1. 首轮仅合并 `feature/infra`（001+002）到 develop，CI 通过后其余角色基于 develop 拉取并开发。
  2. 002/003/004/007 完成后，由 **integration_test** 按依赖顺序合并：先 primitives_tools（005+006），再 wasm_plugin（007→008→009），再 session_cli 的 010，最后 chat 的 011。
  3. 每次合并前：提交方自测通过（build、clippy、单测）；integration_test 合并后跑全量测试与验收清单，失败则反馈给对应角色，修复后重新合并。
- **验收**
  - 由 integration_test 维护验收清单（与 task.md/tasks_details.md 一致），合并到 develop 后执行；问题记录到 issue 或任务看板（INTEGRATION.md 由 status 汇总生成），指派回开发角色。
