# 开发计划规范

本文档定义工程师 Agent 制定开发计划时的**内容要求**、**质量标准**与**自检清单**。  
**案例与模板**：见第四节；最小一屏骨架见 [PLAN_SKELETON.md](./PLAN_SKELETON.md)。

由 [Dispatcher.md](../Dispatcher.md)「四、制定开发计划」引用。

---

## 一、计划必须包含的内容

1. **待完成子项清单**  
   对照当前迭代看板 [TASK_BOARD_002.md](../TASK_BOARD_002.md) 中的子项清单，列出本任务所有待完成子项及当前状态（已完成/待做）。若计划内拆 Phase 导致编号与看板不一致，须加**「看板子项 ↔ 计划子项」映射表**（避免评审与验收时对不上号）。

2. **目标与验收（含功能/步骤的作用与意义、用户故事/用户场景）**  
   - 用一两句话写明「要做出什么」以及验收标准（可运行、可测、门禁通过等）。  
   - 若任务包含多步流程（如加载流程的若干步骤），对**每一步**简要说明：**用户故事/用户场景**（用户在该步骤下会得到什么体验、解决什么问题）、**作用**（做什么）、**意义**（为什么需要、不做的后果）。便于评审与实现时对齐设计意图与用户价值，避免实现偏离或遗漏。  
   - **规格单一来源**：凡与架构/时序/字段语义强相关的任务，须写明**以哪份 openspec 文档现行正文为准**（例如 `context-management.md`）；计划表述与看板表格冲突时**以规格文档为准**并回看板是否需勘误。

3. **对每个子项给出**  
   - **涉及的文件与模块**：改动的源码路径、依赖的需求/设计文档。  
   - **实现思路**：关键数据结构、算法、调用链路（谁调谁、在什么时机）。**跨异步边界**（`async` 运行时、`JoinHandle`、`block_on`/`spawn`、Tokio `Handle`）须在相应子项写清约束，避免在 async 上下文中嵌套阻塞导致死锁或未定义行为。  
   - **依赖的现有接口或需新建的接口**：列出已有 API 与计划新增的 API/回调。  
   - **预期的测试要点**：正常路径、边界与异常（非法输入、失败场景），以及期望的错误表现（信息清晰、不崩溃、可恢复等）。

4. **实施顺序与依赖关系**  
   明确各子项的实施顺序，以及子项之间的依赖（例如先扩展数据结构再实现主流程）。**中大型任务**建议用依赖图（ASCII 或 Mermaid，二选一即可）表达可并行 Phase 与串行关键路径。

5. **风险点与可能的阻塞项**  
   标出对外部接口或未决设计的依赖、与现有模块的兼容性顾虑、以及若无法解决时的备选方案或降级思路。

6. **集成与 E2E 交付要点（条件触发）**  
   若任务涉及**用户可见行为**（如新/改 CLI 子命令、对话流、插件加载与交互等）或变更 **P0/P1 用户故事**相关能力，须写明：
   - 拟更新或新增的 [E2E_SCENARIO_LIBRARY.md](../../openspec/specs/guides/testing/E2E_SCENARIO_LIBRARY.md) 条目（编号或用户意图，可与现有表对照）；**用例名/测试函数名须与仓库内已有或可交付的符号一致**，或明确标注「待新增」并在开发中同步改场景库，禁止长期保留仓库中不存在的虚构用例名。  
   - 计划在 `tests/` 下新增或改动的**集成测试**文件或用例意图（黑盒、`pub` API）；  
   - 计划在 `tests/cli_tests.rs` / `tests/wasmedge_e2e_tests.rs` 中的 `test_user_*` 或 Wasm E2E 覆盖点。  
   **纯内部重构**且无数用户面、无对外 API 行为变更时，可写「不适用」并附一句理由。  
   **全量门禁执行方式**（涉及 §4 时）：须遵守 [INTEGRATION_MERGE_AND_ACCEPTANCE.md](../INTEGRATION_MERGE_AND_ACCEPTANCE.md) 中「后台写日志 + 轮询、禁止短超时前台冒充全量」等约定，不在计划内全文重复，但**须引用**并写明交付前将按该文档执行。

7. **Todo 总表（与正文一一对应，必含）**  
   除上文「待完成子项清单」外，须有**单独可追踪的 Todo 汇总**，覆盖：
   - **流程类**：认领任务与分支、`develop` 同步、按 Dispatcher 的 status/commit/push 节奏、分支侧门禁与标 `PENDING_INTEGRATION` 等（与 [Dispatcher.md](../Dispatcher.md) 一致）；  
   - **实施类**：每个 Phase、每个看板子项或等价粒度，**与正文各章节标题 / 子项编号可双向查找**（总表中每条 todo 在正文有展开；正文每个交付点在总表中有一条 todo，禁止「只在正文散落描述却无汇总」）。  
   **形式**（任选其一或组合，须在计划内写清若双轨如何同步）：  
   - Markdown：顶层 `- [ ] todo_id — 简述` 或表格 `| id | 动作 | 依据/对应章节 |`；  
   - 使用 Cursor 等工具时：可在计划 frontmatter 用 YAML `todos`，并在正文设 **「流程类 Todo（与 YAML 同步）」** 表，避免 UI 与正文各说各话。

## 二、大型 / 规格密集型任务的推荐补充（可选但强烈建议）

当任务跨多 Phase、多模块或强依赖某一架构文档时，在「一」之外可增加下列块（**取精华**：写清决策与基线即可，避免把计划写成第二份规格全书）：

| 块 | 作用 |
| :--- | :--- |
| **当前基线 / 现状与差距** | 列出关键类型、入口函数、旧行为与新目标一行对比，减少「从哪改起」的歧义。 |
| **已定产品 / 技术决策表** | 用户或架构师已拍板的选项（删除/保留某路径、默认值、是否保留 deprecated 等）**表格式列出**；实施阶段不得再悬而未决。 |
| **文档先行与阶段边界** | 若采用「先只改 openspec、后改代码」的两阶段策略，须在计划中写明**阶段范围**与**不包含**的交付物，避免 Agent 越界改 `src/`。 |

**糟粕勿抄**：勿在计划里堆超长 API 清单或与 `TASK_BOARD` 完全重复的 Phase 正文；超大任务可拆为独立 `agents/plan/PLAN_<任务简写>.md`，本规范只定义共性要求。

## 三、好的计划的特征

- **通俗易懂（12 岁原则）**：计划中描述任何事物——概念、模块、流程、bug——都要用 **12 岁小孩也能看懂的语言**先说一遍。具体做法：先用一两句大白话讲清「这是什么、为什么要做」，再展开技术细节；多用短句和比喻，少连续堆砌术语。这条原则同样适用于计划中的问题描述、方案解释和验收标准。（对应 [Constitution.md](../../openspec/specs/Constitution.md) 二.10「通俗易懂原则」）
- **可执行**：读完后能按顺序动手，不需要再猜「先改哪一块」。  
- **可验收**：目标与验收标准清晰，完成后能判断是否算「做完」。  
- **可回溯**：关键步骤有「作用与意义」说明，便于后续维护或需求变更时理解设计。  
- **风险可见**：阻塞与不确定性写清楚，方便用户提前决策或补充信息。  
- **与仓库可对照**：提到的测试名、场景编号、模块路径可在本仓库检索到或明确为待建项。  
- **Todo 完整且与描述匹配**：流程类 + 实施类 todo 已全部列出，且与章节/Phase/看板子项一一对照，无遗漏、无悬空条目。

## 五、计划输出前自检（必做）

Agent **输出计划前**须逐条确认以下项**全部满足，缺一不可**：

- [ ] 每个概念/模块/流程先用一两句大白话说清（12 岁原则），再展开技术细节
- [ ] 列出了全部子项及「已完成/待做」状态（或与看板的映射关系）
- [ ] 写明了总体目标与验收标准（一两句话）
- [ ] 每步/每个子命令写了「用户故事/用户场景」「作用」「意义（不做的后果）」
- [ ] 每个子项列出了「涉及的文件与模块」（含源码路径与依赖的设计文档）
- [ ] 每个子项写了「实现思路」含调用链路（谁调谁、在什么时机）；涉及 async/阻塞的已写清约束
- [ ] 每个子项明确列出了「依赖的现有接口」和「需新建的接口」
- [ ] 每个子项写了「预期测试要点」含边界场景与期望的错误表现
- [ ] 写了实施顺序与子项间依赖关系（中大型含依赖图更佳）
- [ ] 写了风险点，且每个风险有备选方案或降级思路
- [ ] 若任务涉及用户可见行为或 P0/P1 故事：已写清 E2E 场景库、集成测试、E2E 用例的交付要点，且**测试/场景名称与仓库一致或可交付**；若为纯内部重构已标明「不适用」及理由
- [ ] 强规格任务已注明「单一事实来源」文档路径
- [ ] **「一.7 Todo 总表」已写**：流程类 + 实施类条目齐全，且与正文章节/Phase 可一一对照

有任一项未满足，须补全后再输出计划。

## 六、写完计划后的复核（Todo 与正文对照，必做）

计划文本定稿后、交付给用户确认前，**再执行一遍**下列核对（可与第五节合并执行，但**不得省略**）：

- [ ] **Todo → 正文**：总表中每一条 todo（含流程类）在正文中有对应段落、Phase 标题或子节，无「只有勾选无解释」的孤立项。  
- [ ] **正文 → Todo**：正文每个 Phase / 每个交付子项在总表中至少对应一条 todo，无遗漏。  
- [ ] **双轨一致**：若同时使用 YAML `todos` 与 Markdown 表，两处 `id` 与描述一致，并写明「以何者为修改时的同步源」。  
- [ ] **顺序可读**：总表或开篇有「先流程、后实施」或依赖顺序的一句话，与 TASK-20 类长计划中「执行顺序」说明同级清晰。

若复核发现缺口，须回到正文或总表补全后再提交计划。

## 七、Bug 修复类计划：如何描述问题（必读）

当计划的目标是修复 bug 而非新功能时，**问题描述的质量直接决定修复的正确性**。以下规范从 `estimate_context_chars` 水位漂移排查（TASK-17）中提炼，适用于所有涉及"运行时状态不一致"的 bug。

详细排查方法论见 [DEBUG_SPEC.md](./DEBUG_SPEC.md)。

### 7.1 必须包含的问题描述要素

| 要素 | 说明 | 反面 |
| :--- | :--- | :--- |
| **现象** | 用户可见的异常 + 具体数据（比率、字节数、turn 数） | "水位好像不对" |
| **期望** | 正确情况下应该是什么值 | 省略期望，只说"有问题" |
| **量化偏差** | 异常值 - 期望值 = 幽灵/漂移的大小 | 只说"差很多" |
| **对比组** | 至少两组数据：异常 vs 正常（如重启前后） | 只有一个数据点 |

### 7.2 使用逐步场景（Step-by-Step Scenario）描述根因

**不要用抽象语言描述 bug**，而是用一个具体场景，按时间线逐步展示"系统如何一步步走到错误状态"。

**结构模板**：

```markdown
#### 场景设定
（初始状态：关键变量的值，简明的前提条件）

#### Step N: 操作名称（代码位置）
（这一步做了什么）
（代码引用：文件:行号 + 关键代码片段）

    记账 A: xxx → yyy
    记账 B: 不变 / [新增 ...]
    messages: [... 变化 ...]

#### Step N+1: ...
```

**核心原则**：

1. **每一步后列出关键变量的值**——不是只列最终状态，而是每一步都列。读者可以自行验算每一步的变化是否合理。
2. **标注代码位置**——`文件名:行号`，不是"某个函数里"。读者可以直接跳到代码确认。
3. **区分"发生时刻"和"显现时刻"**——bug 的根因可能在 Step 7，但症状在 Step 10 才出现。必须走完到显现那一步，不要在根因处就停下来。

### 7.3 使用"两本账"模型描述状态不一致类 bug

当 bug 的本质是"两个数据结构应该保持一致但脱节了"时，使用"两本账"模型：

```markdown
**记账 A**（`estimate_context_chars`，一个数字）：通过加减法维护
**记账 B**（`messages: Vec<ChatMessage>`，一个列表）：通过 push/drain 维护

正常情况：A ≈ system + sum(B 中每条消息的字符数)
Bug 发生后：A 比上述公式多出 572K，这 572K 在 B 中没有对应物
```

然后在 Step-by-Step 中，每一步同时标注两本账的状态，让读者看到它们在哪一步开始脱节。

### 7.4 用反问句预防误解

当根因的因果链跨越多个时间步骤时，读者很可能在中间某一步产生疑问。在计划中主动回答这些疑问：

**示例**：

> 你可能会问：rebuild 之后 tail 还在 messages 里，实际发给 LLM 的内容和记账 A 不是对得上的吗？
>
> 对，在 L3 重试那一瞬间是对的。但下一轮就不对了——因为 tail 从未被正确加入 `messages`（历史注：原文为"打包进 TurnEntry"，TurnEntry 已在 collapse-to-chatmsg 重构中删除）...

这种写法比让读者自己发现反例后再去计划里找答案要高效得多。

### 7.5 修复方案必须回扣根因

每个 Fix 必须明确说明它修复的是哪个根因，以及修复后两本账如何重新对齐：

```markdown
### Fix B: L3 rebuild 后重算 estimate_context_chars

**修复根因**：根因 2（tail 字符数成为幽灵）
**修复方式**：L3 rebuild 后根据实际 messages 重算记账 A
**对齐公式**：estimate = system + sum(turn_chars) + sum(tail_chars)
**为什么能修**：rebuild 时 system/turns/tail 都是已知的，可以精确计算
```

### 7.6 区分治标和治本，优先治本

修复方案有两种方向，计划中必须标注属于哪种，并说明选择理由：

| 方向 | 含义 | 示例 |
| :--- | :--- | :--- |
| **治标** | 在脱节发生后修补/重算 | L3 后重算 estimate = system + sum(msgs) + tail |
| **治本** | 修正导致脱节的源头操作 | 调整 start_idx 让 tail 被正确加入 `messages`，从根本上消除幽灵 |

**本案例的教训**：初始方案是"L3 rebuild 后重算 estimate"（治标），用户质疑"tail 的 tokens 不就是原来的幽灵吗？"——指出正确做法是校正 `start_idx`（治本）。治标方案虽然能修当前路径，但其他触发路径仍可能产生相同的幽灵。

**在计划中的写法**：

```markdown
### Fix B+C: 校正 start_idx + 移除冗余写入（治本）

**修复根因**：根因 2（幽灵字符）+ 根因 3（User 消息丢失）
**为什么是治本**：不是在幽灵出现后清扫，而是消除幽灵产生的机制——
  让 tail 必定被加入 `messages: Vec<ChatMessage>`，
  从而可被 L2/L3 正常管理。
**为什么可以合并**：两个根因的源头相同——start_idx 设置位置不对。
```

### 7.7 多根因共享同一机制时，描述合并修复的逻辑

当多个根因指向同一个底层机制缺陷时，计划应：

1. **先分别描述**每个根因（保持独立可理解性）
2. **然后指出共同点**："根因 2 和 3 的本质都是 `start_idx` 设置不当"
3. **给出合并方案**：一处改动同时解决多个根因
4. **说明为什么合并更好**：比各自打补丁更少改动点、更少遗漏风险

### 7.8 修复后的对齐验证——用 Step-by-Step 证明修复有效

仅说"修完后两本账对齐了"不够。计划中应**重走一遍场景**，在每一步标注修复后的变量值，证明脱节不再发生：

```markdown
#### 验证：Fix B+C 后的 Step 7（对比修复前）

修复前：start_idx = messages.len() → tail 被跳过 → 幽灵产生
修复后：start_idx = context_tail_start → new_messages 包含完整 tail
         → tail 被正确 push 到 messages
         → 记账 B 包含 tail → L2/L3 可正常扣减 → 无幽灵
```

### 7.9 修复方案必须检查下游兼容性

改动数据的"生产方式"后，所有"消费方"的假设可能失效。计划中须列出检查清单：

```markdown
#### 下游兼容性检查

| 改动 | 消费方 | 检查结论 |
| :--- | :--- | :--- |
| new_messages 含 User | convert_to_llm_format | ✅ 已有 User variant 处理 |
| 移除提前写 User 到 transcript | fold_entries_to_turns | ✅ 从 new_messages 统一写入，顺序不变 |
| start_idx → context_tail_start | L3 tail 提取 | ✅ tail = messages[context_tail_start..] 不变 |
```

### 7.10 自检：问题描述是否充分

- [ ] 有具体数据（数字、比率），不是"好像不对"
- [ ] 有对比组（异常 vs 正常），不是单点观察
- [ ] 有量化偏差（幽灵/漂移的大小），不是"差很多"

## 八、实施阶段经验（从计划到代码）

在计划执行过程中积累的教训，供后续计划制定时参考。

### 8.1 pub(crate) 边界影响测试设计

计划中提到的测试如果涉及 `pub(crate)` 类型（如 `CompactionResult`），必须在计划阶段就区分：

| 目标类型可见性 | 测试位置 | 示例 |
| :--- | :--- | :--- |
| `pub` | `tests/` 集成测试 | `force_drop_oldest_to_target` 的行为 |
| `pub(crate)` | 模块内 `#[cfg(test)]` 单元测试 | `CompactionResult` 的构造与断言 |

**计划中应明确标注**：每个测试子项涉及的类型/函数的可见性，避免实施时才发现编译不过。

### 8.2 门禁顺序：先 Clippy 后 Test

计划的"全量门禁"步骤应写为：

1. `cargo clippy --all-targets -- -D warnings`（先过静态检查）
2. `cargo test`（再跑测试）

Clippy 能发现的问题（未用 import、手写 abs_diff 等）修起来很快，先过 Clippy 能减少测试阶段的无效迭代。

### 8.3 计划中的代码片段应标注文件路径和行号

计划中引用的代码片段必须注明 `文件:行号`，而不是"在某个函数里"。实施时直接跳到对应位置，不需要重新搜索。

**反面**："在 run.rs 的 L3 路径中修改 start_idx"
**正面**："`run.rs:300` — `self.start_idx = messages.len()` 改为 `self.start_idx = tail_start_in_rebuilt`"
