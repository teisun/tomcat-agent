# 开发计划规范

本文档定义工程师 Agent 制定开发计划时的**内容要求**、**质量标准**与**自检清单**。  
**案例与模板**：见第四节；最小一屏骨架见 [PLAN_SKELETON.md](./PLAN_SKELETON.md)。

由 [Dispatcher.md](../Dispatcher.md)「四、制定开发计划」引用。

---

## 一、计划必须包含的内容

1. **待完成子项清单**  
   对照 [TASK_BOARD.md](../TASK_BOARD.md) 中的子项清单，列出本任务所有待完成子项及当前状态（已完成/待做）。若计划内拆 Phase 导致编号与看板不一致，须加**「看板子项 ↔ 计划子项」映射表**（避免评审与验收时对不上号）。

2. **目标与验收（含功能/步骤的作用与意义、用户故事/用户场景）**  
   - 用一两句话写明「要做出什么」以及验收标准（可运行、可测、门禁通过等）。  
   - 若任务包含多步流程（如加载流程的若干步骤），对**每一步**简要说明：**用户故事/用户场景**（用户在该步骤下会得到什么体验、解决什么问题）、**作用**（做什么）、**意义**（为什么需要、不做的后果）。便于评审与实现时对齐设计意图与用户价值，避免实现偏离或遗漏。  
   - **规格单一来源**：凡与架构/时序/字段语义强相关的任务，须写明**以哪份 openspec 文档现行正文为准**（例如 `context-management.md`）；计划表述与看板表格冲突时**以规格文档为准**并回看板是否需勘误。

3. **对每个子项给出**  
   - **涉及的文件与模块**：改动的源码路径、依赖的需求/设计文档。  
   - **实现思路**：关键数据结构、算法、调用链路（谁调谁、在什么时机）。**跨异步边界**（`async` 运行时、`JoinHandle`、`block_on`/`spawn`、Tokio `Handle`）须在相应子项写清约束，避免在 async 上下文中嵌套阻塞导致死锁或未定义行为。  
   - **依赖的现有接口或需新建的接口**：列出已有 API 与计划新增的 API/回调。  
   - **预期的测试要点**：正常路径、边界与异常（非法输入、失败场景），以及期望的错误表现（信息清晰、不崩溃、可恢复等）。

4. **实施顺序与依赖关系**  
   明确各子项的实施顺序，以及子项之间的依赖（例如先扩展数据结构再实现主流程）。**中大型任务**建议用依赖图（ASCII 或 Mermaid，二选一即可）表达可并行 Phase 与串行关键路径。

5. **风险点与可能的阻塞项**  
   标出对外部接口或未决设计的依赖、与现有模块的兼容性顾虑、以及若无法解决时的备选方案或降级思路。

6. **集成与 E2E 交付要点（条件触发）**  
   若任务涉及**用户可见行为**（如新/改 CLI 子命令、对话流、插件加载与交互等）或变更 **P0/P1 用户故事**相关能力，须写明：
   - 拟更新或新增的 [E2E_SCENARIO_LIBRARY.md](../../openspec/specs/guides/testing/E2E_SCENARIO_LIBRARY.md) 条目（编号或用户意图，可与现有表对照）；**用例名/测试函数名须与仓库内已有或可交付的符号一致**，或明确标注「待新增」并在开发中同步改场景库，禁止长期保留仓库中不存在的虚构用例名。  
   - 计划在 `tests/` 下新增或改动的**集成测试**文件或用例意图（黑盒、`pub` API）；  
   - 计划在 `tests/cli_tests.rs` / `tests/wasmedge_e2e_tests.rs` 中的 `test_user_*` 或 Wasm E2E 覆盖点。  
   **纯内部重构**且无数用户面、无对外 API 行为变更时，可写「不适用」并附一句理由。  
   **全量门禁执行方式**（涉及 §4 时）：须遵守 [INTEGRATION_MERGE_AND_ACCEPTANCE.md](../INTEGRATION_MERGE_AND_ACCEPTANCE.md) 中「后台写日志 + 轮询、禁止短超时前台冒充全量」等约定，不在计划内全文重复，但**须引用**并写明交付前将按该文档执行。

7. **Todo 总表（与正文一一对应，必含）**  
   除上文「待完成子项清单」外，须有**单独可追踪的 Todo 汇总**，覆盖：
   - **流程类**：认领任务与分支、`develop` 同步、按 Dispatcher 的 status/commit/push 节奏、分支侧门禁与标 `PENDING_INTEGRATION` 等（与 [Dispatcher.md](../Dispatcher.md) 一致）；  
   - **实施类**：每个 Phase、每个看板子项或等价粒度，**与正文各章节标题 / 子项编号可双向查找**（总表中每条 todo 在正文有展开；正文每个交付点在总表中有一条 todo，禁止「只在正文散落描述却无汇总」）。  
   **形式**（任选其一或组合，须在计划内写清若双轨如何同步）：  
   - Markdown：顶层 `- [ ] todo_id — 简述` 或表格 `| id | 动作 | 依据/对应章节 |`；  
   - 使用 Cursor 等工具时：可在计划 frontmatter 用 YAML `todos`，并在正文设 **「流程类 Todo（与 YAML 同步）」** 表，避免 UI 与正文各说各话。

## 二、大型 / 规格密集型任务的推荐补充（可选但强烈建议）

当任务跨多 Phase、多模块或强依赖某一架构文档时，在「一」之外可增加下列块（**取精华**：写清决策与基线即可，避免把计划写成第二份规格全书）：

| 块 | 作用 |
| :--- | :--- |
| **当前基线 / 现状与差距** | 列出关键类型、入口函数、旧行为与新目标一行对比，减少「从哪改起」的歧义。 |
| **已定产品 / 技术决策表** | 用户或架构师已拍板的选项（删除/保留某路径、默认值、是否保留 deprecated 等）**表格式列出**；实施阶段不得再悬而未决。 |
| **文档先行与阶段边界** | 若采用「先只改 openspec、后改代码」的两阶段策略，须在计划中写明**阶段范围**与**不包含**的交付物，避免 Agent 越界改 `src/`。 |

**糟粕勿抄**：勿在计划里堆超长 API 清单或与 `TASK_BOARD` 完全重复的 Phase 正文；超大任务可拆为独立 `agents/plan/PLAN_<任务简写>.md`，本规范只定义共性要求。

## 三、好的计划的特征

- **可执行**：读完后能按顺序动手，不需要再猜「先改哪一块」。  
- **可验收**：目标与验收标准清晰，完成后能判断是否算「做完」。  
- **可回溯**：关键步骤有「作用与意义」说明，便于后续维护或需求变更时理解设计。  
- **风险可见**：阻塞与不确定性写清楚，方便用户提前决策或补充信息。  
- **与仓库可对照**：提到的测试名、场景编号、模块路径可在本仓库检索到或明确为待建项。  
- **Todo 完整且与描述匹配**：流程类 + 实施类 todo 已全部列出，且与章节/Phase/看板子项一一对照，无遗漏、无悬空条目。

## 五、计划输出前自检（必做）

Agent **输出计划前**须逐条确认以下项**全部满足，缺一不可**：

- [ ] 列出了全部子项及「已完成/待做」状态（或与看板的映射关系）
- [ ] 写明了总体目标与验收标准（一两句话）
- [ ] 每步/每个子命令写了「用户故事/用户场景」「作用」「意义（不做的后果）」
- [ ] 每个子项列出了「涉及的文件与模块」（含源码路径与依赖的设计文档）
- [ ] 每个子项写了「实现思路」含调用链路（谁调谁、在什么时机）；涉及 async/阻塞的已写清约束
- [ ] 每个子项明确列出了「依赖的现有接口」和「需新建的接口」
- [ ] 每个子项写了「预期测试要点」含边界场景与期望的错误表现
- [ ] 写了实施顺序与子项间依赖关系（中大型含依赖图更佳）
- [ ] 写了风险点，且每个风险有备选方案或降级思路
- [ ] 若任务涉及用户可见行为或 P0/P1 故事：已写清 E2E 场景库、集成测试、E2E 用例的交付要点，且**测试/场景名称与仓库一致或可交付**；若为纯内部重构已标明「不适用」及理由
- [ ] 强规格任务已注明「单一事实来源」文档路径
- [ ] **「一.7 Todo 总表」已写**：流程类 + 实施类条目齐全，且与正文章节/Phase 可一一对照

有任一项未满足，须补全后再输出计划。

## 六、写完计划后的复核（Todo 与正文对照，必做）

计划文本定稿后、交付给用户确认前，**再执行一遍**下列核对（可与第五节合并执行，但**不得省略**）：

- [ ] **Todo → 正文**：总表中每一条 todo（含流程类）在正文中有对应段落、Phase 标题或子节，无「只有勾选无解释」的孤立项。  
- [ ] **正文 → Todo**：正文每个 Phase / 每个交付子项在总表中至少对应一条 todo，无遗漏。  
- [ ] **双轨一致**：若同时使用 YAML `todos` 与 Markdown 表，两处 `id` 与描述一致，并写明「以何者为修改时的同步源」。  
- [ ] **顺序可读**：总表或开篇有「先流程、后实施」或依赖顺序的一句话，与 TASK-20 类长计划中「执行顺序」说明同级清晰。

若复核发现缺口，须回到正文或总表补全后再提交计划。
