# 基于规范驱动的 AI Agent 自动化工作流

> 组内技术分享 · 2026-03

---

## 1. 全景：规范是唯一真相源

**一句话**：不是让 AI 自由发挥，而是用一套完整的规范体系约束 Agent 的行为——规范定义"做什么"和"怎么做"，Agent 负责执行，人类负责决策。

```text
┌─────────────────────────────────────────────────────────────────────────┐
│                    规范驱动的 AI Agent 自动化工作流                        │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│   ┌───────────────┐     ┌───────────────┐     ┌───────────────┐       │
│   │   人类（架构师） │     │   规范体系     │     │  AI Agent 团队 │       │
│   │               │     │  (openspec/)  │     │               │       │
│   │ · 写规范       │────►│ · Constitution│────►│ · Tom (工程师)  │       │
│   │ · 拆任务       │     │ · Architecture│     │ · Jerry (工程师)│       │
│   │ · 审计划       │     │ · 编码/测试规范 │     │ · Spike (工程师)│       │
│   │ · 做决策       │     │ · 提交/文档规范 │     │ · Nibbles (QA) │       │
│   └───────┬───────┘     └───────────────┘     └───────┬───────┘       │
│           │                                           │               │
│           │         ┌───────────────────┐             │               │
│           └────────►│  TASK_BOARD.md    │◄────────────┘               │
│                     │  (任务看板)        │                             │
│                     │  唯一状态中枢      │                             │
│                     └─────────┬─────────┘                             │
│                               │                                       │
│                               ▼                                       │
│                     ┌───────────────────┐                             │
│                     │  交付：代码 + 测试  │                             │
│                     │  + 文档 + 进度状态  │                             │
│                     └───────────────────┘                             │
└─────────────────────────────────────────────────────────────────────────┘
```

核心理念：

- **规范先于编码**：先有 Architecture 和 Constitution，再有任务拆解，最后才是 Agent 编码
- **看板是唯一状态中枢**：所有 Agent 从同一个 TASK_BOARD.md 领任务、报状态
- **人类把控关键节点**：规范制定、计划审批、阻塞决策——这三件事人类不放手

## 2. 规范体系：三层架构

**一句话**：从宪法到细则，形成一套完整的"法律体系"——Agent 的一切行为都有据可依。

```text
┌─────────────────────────────────────────────────────────────────────┐
│                          规范体系三层架构                              │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  Layer 1 · 宪法层 (Constitution.md)                                  │
│  ─────────────────────────────────                                  │
│  行为红线 · 安全底线 · 完成定义 · 协作规范                            │
│  「所有 Agent 必须遵守，与宪法冲突的需求须拒绝」                       │
│                          │                                          │
│                          ▼                                          │
│  Layer 2 · 架构层 (Architecture.md + 子文档)                         │
│  ──────────────────────────────────────────                         │
│  系统分层 · 模块职责 · 接口契约 · 技术方案                            │
│  「设计的唯一真相源，实现不得偏离」                                    │
│                          │                                          │
│                          ▼                                          │
│  Layer 3 · 细则层 (guides/)                                         │
│  ──────────────────────────                                        │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐             │
│  │  编码规范     │  │  测试规范     │  │  流程规范     │             │
│  │              │  │              │  │              │             │
│  │ · 架构规范   │  │ · 单元测试    │  │ · Commit 格式 │             │
│  │ · 代码注释   │  │ · 集成测试    │  │ · Status 规范 │             │
│  │ · Rust 惯例  │  │ · E2E 测试   │  │ · 文档规范    │             │
│  └──────────────┘  │ · 场景库     │  │ · 计划规范    │             │
│                    └──────────────┘  └──────────────┘             │
│                                                                     │
│  补充：迭代需求文档 (changes/001-mvp/)                               │
│  ──────────────────────────────────                                │
│  task.md (验收标准) + tasks_details.md (原子子任务) + design.md       │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

每一层的作用：

| 层级 | 文档 | 谁写 | 约束什么 |
|------|------|------|---------|
| 宪法层 | Constitution.md | 人类 | Agent 的行为底线、完成定义、安全红线 |
| 架构层 | Architecture.md + 子文档 | 人类 | 系统设计、模块边界、接口契约 |
| 细则层 | guides/ 下各规范 | 人类 | 编码风格、测试要求、提交格式、文档标准 |
| 迭代需求 | changes/001-mvp/ | 人类 | 本迭代要做什么、验收标准、原子任务拆解 |

**关键设计**：规范全部由人类编写，Agent 只是执行者。Agent 不能修改规范，但如果发现规范有矛盾，必须上报而非自行决策。

**一句话**：三个工程师 Agent 并行开发各自的任务分支，一个集成测试 Agent 负责合并验收——角色分明、职责清晰。

│   │         │   │         │       │         │   （角色相同，        │

│              │   验收        │                                       │
│              │ · 更新看板    │                                       │

| 角色 | 数量 | 职责 | 关键约束 |
|------|------|------|---------|
| **Tom / Jerry / Spike** | 3 | 从看板领任务、读规范、写计划、编码、自测；标 `PENDING` 前须按 [INTEGRATION_MERGE_AND_ACCEPTANCE.md](../../agents/INTEGRATION_MERGE_AND_ACCEPTANCE.md) 完成分支侧集成与 E2E | 一次只领一个任务；计划须经人类确认 |
| **Nibbles** | 1 | 合并到 develop；按同一交付文档复跑命令；全量 review 与补漏；全量验收后更新 DONE | 合并前须人类选择范围；develop 侧独有要求见 [Nibbles.md](../../agents/Nibbles.md) §4（合并后文档与测试） |
| **人类** | 1+ | 写规范、拆任务、审批计划、处理阻塞决策 | 不直接编码，把控关键节点 |

**为什么 Agent 角色相同但要多个？** 并行开发——Tom 做插件系统的同时，Jerry 在做 CLI，Spike 在做 Agent Loop。互不阻塞，各自分支。

## 4. 七步工作流：从领取到交付

**一句话**：每个工程师 Agent 启动后，严格按七步流程执行——不是"自由编码"，而是"按章办事"。

```text
┌─────────────────────────────────────────────────────────────────────┐
│              工程师 Agent 七步工作流 (Dispatcher.md)                   │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  ① 领取任务                                                         │
│  │  读 TASK_BOARD → 找 TODO 且无负责人的最高优先级任务                │
│  │  检查依赖是否全部 DONE                                           │
│  ▼                                                                  │
│  ② 认领任务                                                         │
│  │  写入负责人（Tom/Jerry/Spike）→ 状态改为 DOING                   │
│  ▼                                                                  │
│  ③ 读取上下文                                                       │
│  │  读 Architecture.md → Constitution.md → tasks_details.md         │
│  │  理解模块边界、接口契约、原子子任务与边界场景                      │
│  ▼                                                                  │
│  ④ 制定计划                          ◄──── ★ 人类审批节点             │
│  │  按 PLAN_SPEC.md 输出详细计划                                    │
│  │  含：子项清单 / 文件与模块 / 实现思路 / 测试要点 / 风险点          │
│  │  ────────── 人类确认后方可继续 ──────────                         │
│  ▼                                                                  │
│  ⑤ 开发                                                             │
│  │  同步 develop → 创建 feature/ 分支 → 编码 → 单测 → 写文档        │
│  │  每个子任务完成即提交（禁止囤积）                                  │
│  │  提交格式：what + why + [cov = xx.x%]                           │
│  ▼                                                                  │
│  ⑥ 阻塞处理（如遇到）                                               │
│  │  TASK_BOARD 标记 BLOCKED + 写阻塞原因                            │
│  │  status 文件记录阻塞状态                                         │
│  │  ★ 禁止静默阻塞                                                  │
│  ▼                                                                  │
│  ⑦ 完成任务                                                         │
│     门禁通过（rustfmt / clippy / 单测）                              │
│     更新 status 文件 → 状态改为 PENDING_INTEGRATION                  │
│     推送到远端 → 回到 ① 领取下一个任务                               │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘

    ★ 关键约束：
    · 禁止跳过 ③④ 直接编码
    · 计划未经人类确认，不得进入 ⑤
    · 每个子任务完成即提交，不囤积
    · 遇阻塞必须上报，禁止静默
```

**为什么"计划先行"如此重要？**

这是整个工作流中最关键的设计。传统做法是让 AI 直接编码，结果经常偏离设计意图、遗漏边界场景。规范驱动的做法是：

1. Agent 先读完所有规范和设计文档
2. 输出一份详细计划（含文件路径、调用链路、测试要点、风险项）
3. **人类审批后才开始编码**

这一步把"AI 理解是否正确"的验证前置到了编码之前——修改一份计划文档的成本远低于修改一堆代码。

## 5. 任务状态流转

**一句话**：任务从诞生到完成，经历严格的状态流转——每个状态转换都有明确的触发条件和责任人。

```text
                      工程师认领
  ┌──────┐          ┌──────────┐
  │ TODO │─────────►│  DOING   │
  │      │          │          │
  │ 待认领 │         │ 开发中    │
  └──────┘          └────┬─────┘
                         │
              ┌──────────┼──────────┐
              │ 遇阻塞    │ 自测通过   │
              ▼          │          ▼
        ┌──────────┐     │   ┌──────────────────┐
        │ BLOCKED  │     │   │ PENDING_          │
        │          │     │   │ INTEGRATION       │
        │ 阻塞中    │─────┘   │                    │
        │ (须记录   │ 恢复    │ 等待集成测试        │
        │  原因)    │         │ (工程师标记)         │
        └──────────┘         └─────────┬──────────┘
                                       │
                                       │ Nibbles 验收通过
                                       ▼
                              ┌──────────────┐
                              │     DONE     │
                              │              │
                              │ 已完成        │
                              │ (含集成通过)   │
                              └──────────────┘

  责任分界线：
  ·────────────────────────────────────────────·
  │  工程师负责：TODO → DOING → PENDING_INTEGRATION  │
  │  Nibbles 负责：PENDING_INTEGRATION → DONE        │
  ·────────────────────────────────────────────·
```

**状态治理的设计意图**：

| 设计点 | 目的 |
|--------|------|
| 一次只领一个任务 | 避免多任务切换导致质量下降 |
| BLOCKED 必须记录原因 | 禁止静默阻塞，让人类及时介入 |
| 工程师不能标 DONE | DONE 意味着集成测试也通过了，这由 Nibbles 确认 |
| PENDING_INTEGRATION 不可认领 | 防止其他 Agent 误认领正在集成的任务 |

## 6. 集成验收：Nibbles 的五步流程

**一句话**：工程师在功能分支已按交付文档完成集成与 E2E 后，Nibbles 合并到 develop 仍须按**同一文档**复跑并 review、补漏，全量验收通过后才算真正完成。

```text
┌─────────────────────────────────────────────────────────────────────┐
│              Nibbles 集成验收五步流程                                  │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  Step 1 · 选择合并范围                   ◄── ★ 人类选择              │
│  │  扫描 git 分支 → 列出有未合并提交的功能分支                       │
│  │  人类选择：all / 单分支 / 仅验收不合并                            │
│  ▼                                                                  │
│  Step 2 · 合并前检查                                                 │
│  │  cargo build → cargo clippy → cargo test -j 1 -- --test-threads=1 │
│  │  冲突解决 → 依赖顺序检查                                         │
│  ▼                                                                  │
│  Step 3 · 合并后（顺序与命令见 INTEGRATION_MERGE_AND_ACCEPTANCE）    │
│  │  工程师已在分支完成交付文档 §1→§2→§3→§4；Nibbles 在 develop 上   │
│  │  复跑同一套步骤与命令，并按 Nibbles「合并后文档与测试」做 review/补漏 │
│  ▼                                                                  │
│  Step 4 · 全量验收                                                   │
│  │  构建 → 静态检查 → 全量测试 → CLI 子命令 → 对话模式               │
│  │  → 插件加载/卸载 → 跨平台（如条件具备）                           │
│  ▼                                                                  │
│  Step 5 · 状态更新                                                   │
│     更新 status 文件 → TASK_BOARD 标记 DONE                         │
│     失败项反馈给对应工程师 → 记录到 docs/INTEGRATION.md              │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**集成与 E2E 谁写、何时写？**

- **工程师**在功能分支按 [INTEGRATION_MERGE_AND_ACCEPTANCE.md](../../agents/INTEGRATION_MERGE_AND_ACCEPTANCE.md) 编写/补齐集成与 E2E（黑盒、`pub` API），标 `PENDING_INTEGRATION` 前须通过交付文档 **§4**（全量验收）。
- **Nibbles** 合并后**复跑同一文档**中的命令与顺序，并依 [Nibbles.md](../../agents/Nibbles.md) **§4**（合并后文档与测试）做全量 review 与补漏，**不得**因分支已测而省略复跑。
- 单元测试仍以工程师在开发时维护为主；**行为验收**以交付文档中的集成与 E2E 为准。

## 7. 进度可见性：status 文件

**一句话**：每个功能分支都有一个对应的 status 文件，实时记录进度、阻塞和验收结果——项目状态一目了然。

```text
docs/status/
├── feature-plugin-lifecycle.md     ◄── Tom 的分支进度
├── feature-cli-commands.md         ◄── Jerry 的分支进度
├── feature-long-lived-vm.md        ◄── Tom 的另一个任务
└── develop.md                      ◄── Nibbles 的集成状态

每个 status 文件的结构：

┌─────────────────────────────────────────────────────┐
│ Owner    │ Update Time      │ State              │  │
│ @Tom     │ 2026-03-16 22:30 │ PENDING_INTEGRATION│  │
│ Branch   │ Cov%             │                    │  │
│ feature/ │ 82.3%            │                    │  │
│ long-vm  │                  │                    │  │
├─────────────────────────────────────────────────────┤
│ 已完成子项：                                         │
│ [x] 15.1 结构改造：长寿命运行单元                    │
│ [x] 15.2 RuntimeManager：双键管理                   │
│ [x] 15.3 PluginManager 升级为 session 维度          │
│ [x] ...                                             │
├─────────────────────────────────────────────────────┤
│ 接口变更：                                           │
│ · 新增 __session.waitForEvent 路由                   │
│ · 新增 RuntimeManager / VmActor 类型                 │
├─────────────────────────────────────────────────────┤
│ 阻塞项：无                                           │
└─────────────────────────────────────────────────────┘
```

**设计意图**：

- **分支名即文件名**：`feature/long-lived-vm` → `docs/status/feature-long-lived-vm.md`，一一对应
- **信息可聚合**：所有 status 文件可自动汇总为全局看板（docs/INTEGRATION.md）
- **历史可追溯**：status 文件在 Git 中有完整变更记录

## 8. 合规闭环：自检与复盘

**一句话**：Agent 完成任务前必须过自检清单，违反规范时必须复盘并整改——形成持续改进的闭环。

**开发计划自检（每次输出计划前）**：

```text
  计划输出前，须逐条确认：

  □ 列出了全部子项及「已完成/待做」状态
  □ 写明了总体目标与验收标准
  □ 每步写了「用户故事」「作用」「意义（不做的后果）」
  □ 每个子项列出了「涉及的文件与模块」
  □ 每个子项写了「实现思路」含调用链路
  □ 每个子项明确列出了「现有接口」和「需新建的接口」
  □ 每个子项写了「测试要点」含边界场景
  □ 写了实施顺序与子项间依赖关系
  □ 写了风险点，且每个风险有备选方案

  ────────── 任一项未满足，须补全后再输出 ──────────
```

**完成前自检（每次标记 PENDING_INTEGRATION 前）**：

```text
  □ 已确认当前分支，并已更新对应 status 文件
  □ 门禁通过：rustfmt / clippy / 单测
  □ 技术文档已更新（若有接口变更）
  □ 提交格式合规：what + why + [cov = xx.x%]
  □ 已推送到远端
```

**违规复盘**：当发现 Agent 未遵守规范时，会产出专门的整改文档（如 `CONSTITUTION_REMEDIATION_TASK01.md`），明确列出"哪条没做到 → 原因 → 补做清单"，形成可追溯的改进记录。

---

## 9. 总结：为什么规范驱动有效

```text
┌─────────────────────────────────────────────────────────────────┐
│               规范驱动 vs 自由放养                                │
├──────────────────────────┬──────────────────────────────────────┤
│        自由放养            │          规范驱动                     │
├──────────────────────────┼──────────────────────────────────────┤
│  AI 自由发挥              │  规范定义边界，Agent 按章执行          │
│  结果不可预测             │  输出格式/质量可预期                   │
│  出错后修修补补           │  计划前置 + 自检清单，错误前置拦截      │
│  进度不透明               │  status 文件 + 看板，实时可见          │
│  单人作战                 │  多 Agent 并行，角色分离               │
│  没有沉淀                 │  规范/计划/复盘全部沉淀为文档          │
└──────────────────────────┴──────────────────────────────────────┘
```

**核心经验**：

1. **规范是杠杆**：花 1 小时写规范，节省 Agent 10 小时的返工
2. **计划是最便宜的验证**：审一份计划文档比审一堆代码便宜得多
3. **角色分离是质量保障**：开发和测试不是同一个 Agent，避免"自己测自己"
4. **可见性是信任基础**：status 文件让人类随时知道进度，不需要"追着问"
5. **复盘是进化机制**：违规不可怕，复盘并沉淀整改记录才是关键
