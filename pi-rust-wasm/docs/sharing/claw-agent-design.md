## 1. Claw 核心设计理念

### 1.1 核心架构概览

```text
┌─────────────────────────────────────────────────────────────────┐
│                         Claw 核心架构                            │
├─────────────────────────────────────────────────────────────────┤
│  ┌──────────────┐    ┌──────────────┐    ┌──────────────┐      │
│  │   自进化核心   │    │  Agent Loop  │    │  任务调度器   │      │
│  │  (pi core)   │◄──►│  (执行循环)   │◄──►│ (定时+事件)  │      │
│  └──────┬───────┘    └──────┬───────┘    └──────┬───────┘      │
│         │                   │                    │              │
│         └───────────────────┼────────────────────┘              │
│                             ▼                                   │
│                   ┌──────────────────┐                          │

│                   │ (隔离 · 安全 · 持久) │                       │
│                   └──────────────────┘                          │
```

三大支柱：**自进化**让系统持续变强，**Agent Loop** 是思考-行动-验证的闭环引擎，**任务调度**让 Agent 具备 7×24 自主运行的能力。

### 1.2 自进化核心 (pi core)

**一句话**：AI Agent 不是单次调用的模型，而是能通过插件持续扩展能力、自我优化(自举)的系统。

```text
┌─────────────────────────────────────────────────────────────────────┐
│                        自进化核心 (pi core)                          │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │

│     │  配置规则   │◄─────────│  - 自生成Skill    │            │     │
│     │  反馈调整   │          │  - 能力扩展组件   │             │     │
│     └─────────────┘   加载   └─────────┬────────┘           │     │
│                        使用             │                   │     │
│                                         │                  │     │
│                                         │                  │     │

│                  │         │   Agent Loop    │          │  │     │
│                  │         │                 │          │  │     │

│                  │         │  Take Action    │          │  │     │

│                  │         └────────┬────────┘          │  │     │
│                  │                  │                   │  │     │
│                  │                  │ 观察/学习          │  │     │
│                  │                  ▼                   │  │     │
│                  │         ┌─────────────────┐          │  │     │
│                  │         │   自进化引擎      │          │  │     │

│                  │         │ - 能力缺口分析    │          │  │     │
│                  │         │ - 新技能代码生成   │         │  │     │

│                  │         └────────┬────────┘          │  │     │

│                              │  (Auto-Skills)│  自动入库，循环加载 │
```

核心设计点：

- **一切能力皆插件**：工具、命令、快捷键、UI 组件——都可以通过插件注册，LLM 直接调用
- **用户驱动扩展**：用户编写插件、配置规则、提供反馈，Agent 加载使用
- **Agent 自进化**：Agent 在运行中观察学习，通过自进化引擎分析不足，自动生成新技能代码
- **双向增强**：用户编写的插件和 Agent 自生成的插件共同存入仓库，形成持续进化的闭环
- **动态热更新**：支持运行时生成/重载插件，不中断服务

### 1.3 Agent Loop

**一句话**：Agent 的核心是一个"思考-行动-验证"的循环——不断收集上下文、执行动作、验证结果，直到产出可用产物。

```text
┌─────────────────────────────────────────────────────────────┐
│                       Agent Loop                             │
│                                                              │
│   ┌─────────────┐    ┌─────────────┐    ┌─────────────┐    │
│   │   Gather     │    │    Take     │    │   Verify    │    │
│   │   Context    │───►│   Action    │───►│    Work     │    │
│   │  (收集上下文) │    │  (执行动作)  │    │  (验证结果)  │    │
│   └─────────────┘    └─────────────┘    └──────┬──────┘    │
│          ▲                                      │           │
│          └──────────────────────────────────────┘           │
│                       循环迭代                                │
│                                                              │
│   可选：在 Context 和 Action 之间插入 Planning 步骤           │
│   （需权衡延迟 vs 计划质量）                                  │
└─────────────────────────────────────────────────────────────┘
```

关键洞察：

| 洞察 | 说明 |
|------|------|
| **验证决定上限** | 验证能力越强，Agent 能承担的任务越复杂；验证越弱，越只能停留在演示级别 |
| **验证贯穿全程** | 不只在最后验证——收集阶段就应做部分校验，每次工具调用后触发 hook 做确定性检查 |
| **错误即反馈** | 模型会读取错误输出并调整策略，验证失败不是终点，而是闭环修正的起点 |
| **运行时可干预** | 支持 Steering（注入新指令）、FollowUp（追加追问）、Abort（中断）三种干预 |

**核心不变量**：工具执行结果是唯一会改变下一轮 LLM 输入的数据——无论同步/异步，最终都写回上下文，驱动下一轮推理。

### 1.4 定时任务 + 事件触发

**一句话**：通过事件总线 + 定时调度，Agent 具备 7×24 自主运行的能力——不只是"被问才答"，而是"主动干活"。

```text
┌─────────────────────────────────────────────────────────────┐
│                       任务调度体系                             │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│   ┌─────────────────┐           ┌─────────────────┐        │
│   │    定时任务       │           │    事件触发       │        │
│   │                  │           │                  │        │
│   │  · 定期备份       │           │  · Git PR 通知    │        │
│   │  · 数据同步       │           │  · 消息推送       │        │
│   │  · 健康检查       │           │  · 文件变更       │        │
│   │  · 定期报告       │           │  · 会话开始/结束  │        │
│   └────────┬────────┘           └────────┬────────┘        │
│            │                              │                  │
│            └──────────┬───────────────────┘                  │
│                       ▼                                      │
│              ┌──────────────────┐                            │
│              │  Routines        │                            │
│              │  Engine          │                            │
│              │  (并行执行)       │                            │
│              └──────────────────┘                            │
└─────────────────────────────────────────────────────────────┘
```

任务调度是 Agent 从「被动等用户发指令」升级为「7×24 小时主动执行任务」的核心能力，也是 Claw 架构能落地生产场景的关键设计，通过双模式覆盖绝大多数自动化业务场景。

**定时任务（Cron 驱动）**

- **核心逻辑**：按预设时间周期自动触发任务，解决周期性、确定性的执行需求，任务规则可配置、可追溯。
- **生产实例**：
  - 每日凌晨 2 点自动备份代码仓库、同步业务数据，生成系统健康报告
  - 每 15 分钟巡检服务器/应用状态，异常时自动触发告警与应急修复
  - 每周一自动汇总上周业务数据，生成标准化报表并同步至工作群
- **关键设计**：Claw 的 Routines 引擎支持并行任务、超时中断、失败重试，避免单任务阻塞整个调度体系。

**事件触发（Webhook / 事件监听驱动）**

- **核心逻辑**：监听外部系统/本地环境的事件信号，实时触发对应任务，解决突发性、不确定性的响应需求，做到事件即响应。
- **生产实例**：
  - 监听 Git 仓库 PR 提交事件，自动拉取代码、执行单元测试与代码规范检查，结果直接回写至 PR 评论区
  - 监听客服工单消息，自动查询知识库匹配解决方案，第一时间完成用户回复
  - 监听本地项目文件变更，自动触发文档格式校验、内容备份与版本同步
- **关键设计**：事件触发与 Agent Loop 深度联动，事件发生时自动注入上下文，无需用户额外补充信息即可完成闭环执行。

两种模式可灵活组合，比如「每日定时生成数据报表」+「报表生成完成事件触发同步至共享盘 + 推送通知」，形成完整的自动化工作流，让 Agent 真正实现无人值守的生产级交付。

### 1.5 行动方式：Tools / Bash / Codegen 权衡

**一句话**：Agent 执行动作有三种方式，各有适用场景，关键是根据任务选对工具。

```text
                          Tools
                         ╱     ╲
                  结构化/可控    上下文成本高
                  低风险         可组合性差
                 ╱                        ╲
             Bash ──────────────────── Codegen
          组合能力强                    最灵活
          上下文省                      执行成本最高
          需发现成本                    风险最大
```

| 方式 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| **Tools** | 结构化、高可靠、低风险 | 上下文占用高、可组合性差 | 敏感操作、固定流程 |
| **Bash** | 可组合性强、上下文省 | 发现成本高 | 工具链组合、文件处理 |
| **Codegen** | 最灵活、动态脚本 | 执行成本高、风险大 | 复杂逻辑、API 组合 |

**关键洞察**：Bash 被严重低估——它能把工具输出沉淀为文件，成为 Agent 的"外部记忆"与可复查证据。多个 Bash 命令可自由组合，不像 Tool 每个都是独立孤岛。

## 2. 构建生产级 AI Agent

### 2.1 标准：交付可用产物

**一句话**：Agent 不是"更会答题的模型"，而是一个由模型、工具、文件系统与验证机制构成的系统——标准是"能持续交付可用结果"。

> "许多 agent 的失败不是因为模型不强，而是缺少可靠的上下文收集与验证机制。"
> —— Thariq Shihipar, Claude Code 团队

```text
  演示级 Agent ─────────────────► 生产级 Agent
       │                                │
       ▼                                ▼
    单次成功                       持续可靠交付
    无验证                         验证贯穿全流程
    无记忆                         长期记忆系统
    无安全                         多层安全防护
    手动触发                       自主运行 (7×24)
```

从演示到生产的四个跨越：**验证机制**、**记忆系统**、**安全架构**、**自主调度**。

---

### 2.2 如何构建

**一句话**：构建 Agent 的关键不是选最强的模型，而是设计好工具体系、上下文管理和反馈回路。

**Skills = 渐进式上下文披露**：

| 方式 | 问题 |
|------|------|
| 全塞 prompt | token 浪费，干扰推理 |
| **Skills 按需加载** | 需要时才加载专业流程，节省上下文 |

**文件系统是最被低估的基础设施**：

- 工具结果落文件、返回文件路径 → 后续可搜索/复查/验证
- 文件系统本身就是 Agent 的"外部记忆"
- 长任务中保持可复查、可迭代的状态

**Todo list 驱动复杂任务**：Agent 自维护 todo 清单，逐步勾选，让长任务有迹可循。

---

### 2.3 记忆系统

**一句话**：生产级 Agent 需要跨对话的长期记忆，最务实的方案是"Markdown 文件 + 混合搜索 + 时序衰减"。

**两层记忆结构**：

```text
~/.agent/workspace/
├── memory/
│   ├── 2026-03-19.md      ◄── 每日日志（只追加，自动加载今天+昨天）
│   ├── 2026-03-18.md
│   ├── ...
│   └── archive/            ◄── 月度归档
└── MEMORY.md               ◄── 长期记忆（手动维护，仅私聊加载）
```

```text
┌─────────────────────────────────────────────────────────┐
│                    记忆系统架构                            │
├─────────────────────────────────────────────────────────┤
│                                                           │
│  ┌─────────────────────────────────────────────────┐   │
│  │              存储层 (Markdown-first)            │   │
│  │                                                   │   │
│  │  ┌──────────────────┐    ┌──────────────────┐  │   │
│  │  │  短期记忆        │    │  长期记忆        │  │   │
│  │  │  (每日日志)      │    │  (MEMORY.md)     │  │   │
│  │  │                  │    │                  │  │   │
│  │  │ 2026-03-19.md   │    │ - 用户偏好       │  │   │
│  │  │ 2026-03-18.md   │    │ - 常用配置       │  │   │
│  │  │ ...              │    │ - 重要决策       │  │   │
│  │  │                  │    │                  │  │   │
│  │  │ 仅追加不修改     │    │ 仅私聊加载       │  │   │
│  │  │ 自动加载今+昨    │    │ 不参与时序衰减   │  │   │
│  │  └──────────────────┘    └──────────────────┘  │   │
│  └─────────────────────────────────────────────────┘   │
│                           │                               │
│                           ▼                               │
│  ┌─────────────────────────────────────────────────┐   │
│  │              搜索层 (混合搜索)                   │   │
│  │                                                   │   │
│  │  ┌──────────────┐    ┌──────────────┐           │   │
│  │  │  向量搜索    │    │  BM25 关键词  │           │   │
│  │  │  (语义理解)   │    │  (精确匹配)    │           │   │
│  │  └──────┬───────┘    └──────┬───────┘           │   │
│  │         │                     │                  │   │
│  │         └──────────┬──────────┘                  │   │
│  │                    ▼                             │   │
│  │           结果融合 + 智能排序                    │   │
│  │                                                   │   │
│  │  - MMR 多样性重排 (避免单一角度重复)            │   │
│  │  - 时序衰减 (新记忆权重高)                       │   │
│  └─────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
```

**核心优势/理念**：

- **Markdown-first**
- 可直接编辑、Git 管理、跨机器拷贝
- 混合搜索互补单一方案的短板
- 时序衰减解决旧记忆干扰问题

**为什么用 Markdown 文件而不是向量数据库？**

- **可见**：直接打开看/改，记错了改一行就好
- **可版本管理**：放 Git 里，看 diff，随时回滚
- **可迁移**：`cp -r` 到新机器即可
- **零依赖**：不需要额外的数据库服务

**混合搜索解决单一方案的盲区**：

```text
搜索 "REDIS_PASSWORD"                搜索 "之前那个性能问题"
         │                                      │
         ▼                                      ▼
  ┌─────────────┐                        ┌─────────────┐
  │ 向量搜索  ✗  │ 语义上不像              │ BM25     ✗  │ 没有"性能"这个词
  │ BM25     ✓  │ 精确匹配到了            │ 向量搜索  ✓  │ 理解"性能"≈"响应慢"
  └─────────────┘                        └─────────────┘

  混合搜索 = 向量(0.6~0.7) + BM25(0.3~0.4) ──► 两种场景都能覆盖
```

**时序衰减**：旧记忆权重指数递减，新记忆优先

```text
权重
1.0 ┤●
    │ ╲
0.5 ┤   ●  ◄── halfLife = 14 天
    │     ╲
0.25┤       ●
    │         ╲
0.01┤            ·  ·  ·  ·
    └──┬───┬───┬───┬───┬──── 天数
       0   14  28  42  90

例外：MEMORY.md 等永久记忆不衰减
```

**MMR 多样性重排**：搜索"数据库优化"时，避免 10 条结果都在讲索引——MMR 会打散结果，让索引、连接池、查询重写等不同方向都出现。

### 2.4 安全架构

**一句话**：25,000+ Agent 实例在公网裸奔，密码/API Key 直接暴露给 LLM——这是当前 Agent 生态的最大系统性风险。

**问题全景**：

```text
┌─ 当前主流 Agent 架构的安全现状 ────────────────────────────────┐
│                                                                │
│  用户凭证 ──直接传入──► LLM Provider ──可被访问──► 公司员工     │
│                                                                │
│  恶意 Skill ──提示注入──► Agent ──执行命令──► 窃取数据          │
│                                                                │
│  25,000+ 公开实例在公网裸奔（被称为 "security dumpster fire"）  │
│                                                                │
│  攻击面：认证绕过 / 命令注入 / API Key 泄露 / 提示词攻击       │
└────────────────────────────────────────────────────────────────┘
```

**四层纵深防御**（综合 IronClaw / ZeroClaw / TuanziGuardianClaw）：

```text
┌─────────────────────────────────────────────────────────────┐
│                    安全纵深防御架构                            │
├─────────────────────────────────────────────────────────────┤
│                                                               │
│  ┌───────────────────────────────────────────────────────┐ │
│  │  Layer 4: 可信执行环境 (TEE)                          │ │
│  │           硬件级隔离，云服务商也无法访问                │ │
│  └───────────────────────────────────────────────────────┘ │
│                              │                                │
│  ┌───────────────────────────▼───────────────────────────┐ │
│  │  Layer 3: 加密凭证保险库                                │ │
│  │           AES-256-GCM 加密存储                          │ │
│  │           凭证绑定域名策略，LLM 无法访问                 │ │
│  │           仅在网络边界注入                               │ │
│  └───────────────────────────────────────────────────────┘ │
│                              │                                │
│  ┌───────────────────────────▼───────────────────────────┐ │
│  │  Layer 2: WASM 沙箱隔离                                │ │
│  │           第三方工具/AI 生成代码在独立容器运行          │ │
│  │           破坏范围严格限制在沙箱内                      │ │
│  └───────────────────────────────────────────────────────┘ │
```

**五级权限分级 + Capability Token**：

| Level | 允许操作 | 策略 |
|-------|---------|------|
| L0 | 文本处理、逻辑推理、格式整理 | 直接放行 |
| L1 | 读取用户明确请求的特定文件 | 系统目录/密钥文件受限 |
| L2 | API 调用、程序执行、包安装 | 每一步需用户确认 |
| L3 | Shell 命令、系统配置、后台进程 | 必须获得用户明确批准 |
| L4 | root 命令、大规模读取、环境变量导出 | 除非用户反复坚持，否则一律阻断 |

**提示词注入防御链路**：

```text
外部输入（网页/邮件/文档）
       │
       ▼
  ┌─────────────────────────────┐
  │ 关键词拦截                    │  "ignore previous instructions"
  │ "bypass security" 等         │──► 直接阻断 + 日志 + 告警
  └──────────┬──────────────────┘
             │ 通过
             ▼
  ┌─────────────────────────────┐
  │ 行为意图审查                  │  不看输入内容，审查动作意图
  │ · 向未知域名发送本地数据？     │──► 标记高风险
  │ · 导出环境变量？              │
  │ · Base64 编码后外传？         │
  └──────────┬──────────────────┘
             │ 通过
             ▼
  ┌─────────────────────────────┐
  │ Capability Token 检查         │  无对应权限 Token
  │ 操作需持有对应权限 Token       │──► 直接拒绝
  └─────────────────────────────┘
```

**核心安全原则**：

1. **最小权限**：只授予必要的最小权限
2. **显式许可**：敏感操作必须用户确认
3. **零信任**：Skill 之间互不信任，互相隔离
4. **数据主权**：用户对数据拥有完全控制权
5. **安全优先于便利**：冲突时安全优先
