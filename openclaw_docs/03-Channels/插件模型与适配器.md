# Channels 插件模型与适配器

## 零、先用大白话

ChannelPlugin 像 **七巧板底板**。  
你必须拼上几块：**配置、出站** 往往少不了。  
**登录、配对、群 @** 这几块，有的渠道要，有的可以偷懒用默认。

**这一节你会学到**：每块适配器干嘛；扩展仓库目录长啥样。

---

**设计思想**：每个 Channel 实现 ChannelPlugin，提供一组适配器（config、gateway、outbound、pairing、security、groups、mentions 等）。适配器职责清晰，Channel 按需实现，未实现的由默认或跳过处理。

## ASCII 核心四图

### 1) 结构图

```text
ChannelPlugin
 ├── config / security
 ├── gateway（WS 相关钩子）
 ├── outbound（sendText/sendMedia）
 ├── pairing / groups / mentions
 └── …（按需实现，缺省走默认）
```

### 2) 调用流图

```text
加载 manifest
  -> 实例化 plugin
      -> 注册各适配器到 registry
          -> Gateway 装配时按名称取适配器
              -> 未实现 -> no-op 或内置 fallback
```

```text

```

### 4) 数据闭环图

```text

        |
        v

        |

```

## 一、适配器职责

| 适配器 | 职责 |
|--------|------|
| config | listAccountIds、resolveAccount、allowFrom、describeAccount |
| gateway | startAccount、stopAccount、loginWithQr、logout |
| outbound | deliveryMode、sendText、sendMedia、resolveTarget、chunker |
| pairing | 配对码生成、approve、存储 |
| security | dmPolicy、allowlist 校验 |
| groups | resolveRequireMention、resolveGroupIntroHint |
| mentions | 解析 @ 提及、提取被提及用户 |
| streaming | 流式发送、chunk 策略 |
| directory | 联系人、群组目录 |

---

## 二、Extension 实现模式

- **目录结构**：`extensions/<channel>/` 下常见 `src/channel.ts`、`index.ts`、**`openclaw.plugin.json`**（以具体扩展为准）。  
- **channel.ts**：导出 `ChannelPlugin` 对象，实现上述适配器。  
- **`openclaw.plugin.json`**：`id`、`main`、`channels` 等，供插件发现逻辑（如 **`discoverOpenClawPlugins`**）扫到。

---

## 三、各 Channel 差异

- **WhatsApp**：Baileys，QR 登录，JID 格式。
- **Telegram**：grammY，Bot Token，chatId。
- **Slack**：Bolt，Bot/App Token，channelId。
- **Discord**：discord.js，Bot Token，guildId、channelId。
- **Signal**：signal-cli，UUID 配对。

适配器实现因协议不同而各异，但接口统一。

## 常见误会

- **误会**：没实现的适配器会报错退出。**正解**：很多是 **可选**；会走默认或 no-op（以类型与运行时检查为准）。  
- **误会**：扩展渠道和内置渠道两套 API。**正解**：**同一套** `ChannelPlugin` 类型；差别在包放哪。  
- **误会**：我只改 `SKILL.md` 就能让 Telegram 变聪明。**正解**：技能是 Agent 侧；Channel 只管 **消息怎么进出**。
