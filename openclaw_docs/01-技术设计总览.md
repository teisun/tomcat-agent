## 零、先用大白话

把 OpenClaw 想成 **「机场塔台 + 跑道」**：塔台是 **Gateway**（谁连着、谁起飞降落）；每条消息是一架飞机，要先 **路由** 到正确的 **Agent** 跑道，再落地（回复）回到乘客用的 App。配置像塔台的 **规则手册**，存在 **`~/.openclaw/openclaw.json`**，可以热更新，不用每次都重启整个机场。

## 一、设计思想

OpenClaw 遵循以下核心设计原则：

1. **本地优先**：配置与会话数据默认在 **`~/.openclaw/`**（可用 `OPENCLAW_STATE_DIR` 覆盖）与 workspace；Gateway 默认 loopback，可选 Tailscale / SSH 暴露。
2. **Gateway 仅作控制面**：协调会话、通道、配置、Cron、事件；Agent、Channel、工具在各模块内执行。
3. **多通道统一收件箱**：各渠道以 ChannelPlugin 注册，入站经 auto-reply，路由解析 session key 后交给 Agent。
4. **Agent 进程内队列化**：Pi Agent 嵌入式运行，同一 session 经 session lane 串行（`enqueueCommandInLane`）。
5. **配置单文件与热更**：单一 **`OpenClawConfig`**，默认文件 **`openclaw.json`**；Gateway `config-reload` 监听变更。

## 二、架构图与数据流

### 2.1 主架构图（ASCII）

```text
Channels -----> Gateway -----> Routing -----> Agents
   ^               |               |              |
   |               |               |              +--> Memory（查笔记）
   |               |               |              +--> Media（看图听音）
   |               v               v              |
   +-------- Outbound / WS 事件 <--+--------------+
Infra（outbound、tailscale、重试…）给 Gateway / Channels 打下手
```

**这一节你会学到**：塔台（Gateway）不推理；推理在 Agents；Memory/Media 是外挂能力。

### 2.2 核心数据流

**入站消息流**：

```text
Channel 入站 → monitorWebInbox / 各 channel monitor
  → 路由解析 (resolveAgentRoute) → Session Key
  → handleCommands（Chat 命令）或 getReplyFromConfig（Agent 回复）
  → runEmbeddedPiAgent（Pi 嵌入式）
  → 工具调用 / Memory 检索 / Media 理解
  → deliver-reply → Channel Outbound
```

**关键路径**（以当前 `main` 为准，文件名可能继续演进）：

- **Monitor（以 Web 渠道为例）**：重逻辑由插件运行时懒加载（如 **`src/plugins/runtime/runtime-web-channel-plugin.ts`** 暴露的 `monitorWebChannel` / `monitorWebInbox`）；具体 handler 与编排仍落在 **`src/auto-reply/`** 树内。
- **路由**：**`src/routing/resolve-route.ts`** 的 **`resolveAgentRoute`** → `sessionKey`、`agentId`。
- **命令 vs 回复**：**`src/auto-reply/reply/commands-core.ts`** 的 **`handleCommands`**；否则 **`getReplyFromConfig`**（`src/auto-reply/reply/get-reply.ts` 一带）→ **`runPreparedReply`**（`src/auto-reply/reply/get-reply-run.ts`）→ **`runEmbeddedPiAgent`**（**`src/agents/pi-embedded.ts`** / **`src/agents/pi-embedded-runner/run.ts`**）。
- **出站**：**`src/infra/outbound/`** 的投递逻辑与 **`deliveryMode`**（direct/gateway/hybrid）。

### 2.3 Auto-reply 端到端流程

```text
入站消息
  → 各渠道 monitor / inbox（实现分散在对应 channel 与 `src/auto-reply/`）
  → inbound debounce、group-gating、mention 检查
  → resolveAgentRoute 得到 sessionKey、agentId
  → handleCommands：若为 Chat 命令，直接处理并返回
  → 否则：getReplyFromConfig
    → finalizeInboundContext（媒体、模板）
    → applyMediaUnderstanding（图/音/视频理解）
    → resolveReplyDirectives（指令解析）
    → runPreparedReply → runEmbeddedPiAgent
    → 流式回调：typing、onPartialReply、onBlockReply
    → memory-flush（回合后 compaction）
  → deliverWebReply / 各 channel 的 send
```

## 三、模块边界与依赖

| 模块 | 职责 | 消费方 | 依赖 |
|------|------|--------|------|
| Config | 加载、校验、合并 **OpenClawConfig** | Gateway、CLI、Channels、Agents、所有模块 | - |
| Gateway | 控制面、WS、Methods、Cron、健康、侧车 | CLI、UI、Channels、Agents、Nodes | Config、Plugins |
| Channels | 入站/出站、配对、安全 | Gateway、auto-reply | Config、Infra/outbound、Pairing |
| Routing | 解析 session key、agent 路由 | auto-reply、Gateway | Config/sessions |
| Agents | Pi 嵌入式、工具、技能 | CLI、auto-reply、Cron、Hooks | Config、Gateway、Memory、Channels |
| Memory | 向量/混合检索、索引、embedding | Agents、auto-reply、Hooks | Config、Media |
| Media-understanding | 图/音/视频描述与转录 | auto-reply、Agents | Config |
| Plugins | 加载、slots、extensions | Gateway、Channels | Config |
| Hooks | 事件钩子 | auto-reply、Gateway | Config |

**配置加载**：`src/config/io.ts` 的 `loadConfig` 通过 **`src/config/paths.ts`** 解析路径；默认 **`~/.openclaw/openclaw.json`**（或 `OPENCLAW_CONFIG_PATH` / 兼容旧 `clawdbot.json`）；Gateway 的 `startGatewayConfigReloader` 监听文件变化并热更。

## 五、关键实现决策

1. **JSON-RPC 风格 Methods**：`listGatewayMethods` 注册；`message-handler` → `coreGatewayHandlers`；事件 `GATEWAY_EVENTS`。
2. **ChannelPlugin 适配器契约**：config、gateway、outbound、pairing、security 等；内置在 `src/channels/`，扩展在 `extensions/*`，清单 **`openclaw.plugin.json`**。
3. **Session lane 串行**：`enqueueCommandInLane(sessionLane, ...)`。
4. **Embedding 批处理与混合检索**：Memory 批量 embedding；`searchVector` + `searchKeyword` hybrid。

## 六、常见误会

- **误会**：Gateway 里跑 Pi，所以 Gateway 很重。**正解**：Gateway 协调连接与 RPC；Pi 嵌入式在 **Agent 执行路径**里跑，模块边界见上表。  
- **误会**：`auto-reply` 只是一个文件。**正解**：它是一条 **管线**（目录 `src/auto-reply/`），入站、命令、回复拼装都在这条链上。  
- **误会**：路由只看「谁发的微信」。**正解**：路由综合 **通道、账号、会话键、绑定规则**；细节见 [05-Sessions与Routing.md](05-Sessions与Routing.md)。
