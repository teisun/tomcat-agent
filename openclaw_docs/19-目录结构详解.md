# OpenClaw 目录结构详解

本文档详细说明 OpenClaw 项目的两类目录结构：**仓库源码目录**（开发者视角）和**运行时状态目录**（`~/.openclaw/`，部署/运行时产生的数据）。

---

## 第一部分：仓库源码目录

OpenClaw 是一个 pnpm monorepo，采用 TypeScript (ESM) 开发，Node >= 22。

### 1.1 顶层目录总览

```
openclaw/
├── src/                  # 核心源码（约 54 个子模块）
├── extensions/           # 独立插件包（workspace packages）
├── apps/                 # 客户端应用（Android/iOS/macOS）
├── ui/                   # Web 控制面板（Vite）
├── packages/             # npm 分发包
├── docs/                 # Mintlify 文档站源码
├── skills/               # 技能包
├── scripts/              # 构建/CI/运维脚本
├── vendor/               # 第三方依赖
├── test/                 # E2E 与集成测试
├── test-fixtures/        # 测试夹具
├── git-hooks/            # 本地 Git 钩子
├── patches/              # pnpm 补丁（占位）
├── assets/               # 静态资源
├── .github/              # CI/CD、issue templates
├── .pi/                  # Pi 定制（扩展、prompts）
├── .agents/              # 维护者/流程说明
├── .agent/               # Agent 工作流文档
└── .vscode/              # 编辑器推荐设置
```

## 第二部分：运行时状态目录 (~/.openclaw/)

当 OpenClaw Gateway 运行时，会在用户主目录下创建 `~/.openclaw/` 状态目录。可通过环境变量 `OPENCLAW_STATE_DIR` 覆盖。

### 2.1 目录总览

```

├── openclaw.json                      # 主配置文件
├── agents/                            # 各 agent 的运行时状态
│   └── <agentId>/
│       ├── agent/                     # 身份与凭据
│       │   ├── auth-profiles.json     # 认证配置（API key/OAuth）
│       │   └── models.json            # 模型配置
│       ├── sessions/                  # 会话记录
│       │   ├── sessions.json          # 会话索引
│       │   └── <sessionId>.jsonl      # 会话 transcript
│       └── qmd/                       # QMD 后端状态
│           ├── xdg-config/            # QMD 配置
│           └── xdg-cache/

├── workspace/                         # 默认 agent 的工作区

│   ├── SOUL.md                        # 人格/灵魂文件
│   ├── TOOLS.md                       # 工具说明
```

#### `agents` — Agent 配置详解

```typescript
type AgentsConfig = {
  defaults?: AgentDefaultsConfig;  // 所有 agent 的默认值
  list?: AgentConfig[];            // agent 列表
};

type AgentConfig = {
  id: string;                      // agent ID
  default?: boolean;               // 是否为默认 agent
  name?: string;                   // 显示名
  workspace?: string;              // 工作区路径（覆盖默认）
  agentDir?: string;               // 运行时状态目录（覆盖默认）
  model?: AgentModelConfig;        // 模型配置（primary + fallbacks）
  skills?: string[];               // 技能白名单（空数组 = 禁用全部）
  memorySearch?: MemorySearchConfig; // 记忆检索配置
  identity?: IdentityConfig;       // 身份/人格
  sandbox?: AgentSandboxConfig;    // 沙箱配置
  tools?: AgentToolsConfig;        // 工具配置
  runtime?: AgentRuntimeConfig;    // 运行时类型：embedded 或 acp
  humanDelay?: HumanDelayConfig;   // 模拟人类回复延迟
  heartbeat?: HeartbeatConfig;     // 心跳/定时触发
  groupChat?: GroupChatConfig;     // 群聊配置
  subagents?: {                    // 子 agent 配置
    allowAgents?: string[];        // 允许生成的子 agent ID
    model?: AgentModelConfig;      // 子 agent 默认模型
  };
};
```

`agents.list` 中的每个 agent 会决定：
- 其 `agentDir` 落在 `~/.openclaw/agents/<id>/agent/`
- 其 `workspace` 落在 `~/.openclaw/workspace-<id>/`（非默认 agent）
- 其 `memorySearch` 索引落在 `~/.openclaw/memory/<id>.sqlite`

#### `gateway` — 网关配置详解

```typescript
type GatewayConfig = {
```

### 2.3 各目录详解

#### `agents/<agentId>/agent/` — 运行时身份

存放系统内部状态，**不应手动编辑**。

- **`auth-profiles.json`**：认证凭据（API key、OAuth token 等），由 `resolveAuthStorePath()` 解析路径（见 `src/agents/auth-profiles/paths.ts`）。
- **`models.json`**：模型配置，由 `listAgentModelsJsonPaths()` 扫描（见 `src/secrets/storage-scan.ts`）。

默认路径由 `resolveAgentDir()` 决定：

```typescript
// src/agents/agent-scope.ts
export function resolveAgentDir(cfg: OpenClawConfig, agentId: string) {
  const id = normalizeAgentId(agentId);
  const configured = resolveAgentConfig(cfg, id)?.agentDir?.trim();
  if (configured) {
    return resolveUserPath(configured);
  }
  const root = resolveStateDir(process.env);
  return path.join(root, "agents", id, "agent");
}
```

**重要约束**：禁止两个 agent 共享同一个 `agentDir`，否则会导致认证和 session 冲突（见 `src/config/agent-dirs.ts`）。

#### `agents/<agentId>/sessions/` — 会话记录

- **`sessions.json`**：会话索引文件
- **`<sessionId>.jsonl`**：每个会话的完整 transcript（JSON Lines 格式）
- 支持 topic 分片：`<sessionId>-topic-<topicId>.jsonl`

路径由 `resolveAgentSessionsDir()` 决定（见 `src/config/sessions/paths.ts`）：

```typescript
function resolveAgentSessionsDir(agentId?: string): string {
  const root = resolveStateDir(env, homedir);
  const id = normalizeAgentId(agentId ?? DEFAULT_AGENT_ID);
  return path.join(root, "agents", id, "sessions");
}
```

#### `agents/<agentId>/qmd/` — QMD 后端

当 memory 后端配置为 `"qmd"` 时，QMD 的 XDG 配置和缓存目录按 agent 隔离（见 `src/memory/qmd-manager.ts`）：


#### `workspace/` / `workspace-<agentId>/` — Agent 工作区

这是**用户可见、可直接编辑**的目录，是 agent 的"工作目录"。

工作区文件与目录（在 `src/agents/workspace.ts` 和 `src/agents/skills/workspace.ts` 中定义）：

| 文件/目录 | 用途 |
|------|------|
| `AGENTS.md` | Agent 引导指令 |
| `SOUL.md` | 人格/角色定义 |
| `TOOLS.md` | 工具使用说明 |
| `IDENTITY.md` | 身份定义 |
| `USER.md` | 用户画像 |
| `HEARTBEAT.md` | 心跳/定时任务配置 |
| `BOOTSTRAP.md` | 启动引导 |
| `MEMORY.md` / `memory.md` | 长期记忆主文件 |
| `memory/` | 记忆子目录（`YYYY-MM-DD.md` 按日存储） |
| **`skills/`** | **工作区技能目录**（优先级最高，覆盖所有其它来源的同名技能） |
| **`.agents/skills/`** | **项目级技能目录**（优先级仅次于 `skills/`） |

路径解析规则（`resolveAgentWorkspaceDir()`）：
1. 若 agent 配置了 `workspace` 字段 -> 使用配置值
2. 若是默认 agent -> `~/.openclaw/workspace`（或 `~/.openclaw/workspace-<profile>` 若设置了 `OPENCLAW_PROFILE`）
3. 若是非默认 agent 且未配置 -> `~/.openclaw/workspace-<agentId>`

#### `memory/<agentId>.sqlite` — 向量检索索引

这是从工作区 Markdown 文件构建出来的**派生索引**，用于加速语义检索。

路径由 `resolveStorePath()` 决定（见 `src/agents/memory-search.ts`）：

```typescript
function resolveStorePath(agentId: string, raw?: string): string {
  const stateDir = resolveStateDir(process.env, os.homedir);
  const fallback = path.join(stateDir, "memory", `${agentId}.sqlite`);
  if (!raw) {
    return fallback;
  }
  const withToken = raw.includes("{agentId}")
    ? raw.replaceAll("{agentId}", agentId) : raw;
  return resolveUserPath(withToken);
}
```

可通过 `memorySearch.store.path` 配置项自定义，支持 `{agentId}` 占位符。

#### `credentials/` — OAuth 凭据

Web provider 的 OAuth 凭据存储目录，运行 `openclaw login` 时写入。

#### `subagents/runs.json` — 子 Agent 注册表

记录所有子 agent 运行实例的注册信息，路径由 `resolveSubagentRegistryPath()` 决定。

#### `skills/` — 托管技能（managed skills）

运行时状态目录下的 `skills/` 用于存放**托管技能**，即通过 Gateway RPC（`skills.install`、`skills.update`）或其它方式安装到本机的技能。每个技能是一个子目录，核心文件为 `SKILL.md`。

路径由 `loadSkillEntries()` 中的默认值决定（见 `src/agents/skills/workspace.ts`）：

```typescript
const managedSkillsDir = opts?.managedSkillsDir ?? path.join(CONFIG_DIR, "skills");
// CONFIG_DIR = ~/.openclaw/
```

托管技能在 6 层技能优先级中排第 3 位（详见附录 Q3）。

#### `tools/` — 技能工具运行时数据

每个技能可能附带可执行工具（如 CLI 二进制），其运行时文件按技能 key 的安全哈希值分目录存放。

路径由 `resolveSkillToolsRootDir()` 决定（见 `src/agents/skills/tools-dir.ts`）：

```typescript
export function resolveSkillToolsRootDir(entry: SkillEntry): string {
  const key = resolveSkillKey(entry.skill, entry);
  const safeKey = safePathSegmentHashed(key);
  return path.join(resolveConfigDir(), "tools", safeKey);
}
// 结果：~/.openclaw/tools/<skillKey-hash>/
```

## 第三部分：多 Agent 隔离机制

在多 agent 场景下，OpenClaw 的目录隔离策略如下：

### 3.1 隔离矩阵

| 目录 | 是否按 agent 隔离 | 能否共享 | 说明 |
|------|-------------------|---------|------|
| `agents/<id>/agent/` | 隔离 | **禁止共享** | 代码强制校验唯一性，冲突会报 `DuplicateAgentDirError` |
| `agents/<id>/sessions/` | 隔离 | 禁止 | 与 `agentDir` 同级，随 agent 隔离 |
| `agents/<id>/qmd/` | 隔离 | 禁止 | QMD 状态按 agent 分目录 |
| `workspace(-<id>)/` | 默认隔离 | **允许显式共享** | 多个 agent 可配置同一个 `workspace` 路径 |
| `workspace(-<id>)/skills/` | 随 workspace | 随 workspace | 工作区技能随 workspace 共享或隔离 |
| `memory/<id>.sqlite` | 隔离 | 允许自定义 | 默认按 agent 分文件，可通过 `store.path` 自定义 |
| `skills/` | **全局** | 所有 agent 共享 | 托管技能目录，不按 agent 隔离 |
| `tools/` | **全局** | 所有 agent 共享 | 技能工具目录，按技能 key 哈希分目录 |

### 3.2 设计原则

目录结构遵循**按职责分离**而非"按 agent 聚合一切"的原则：

```
~/.openclaw/
├── agents/       # 身份与会话（系统状态，禁止共享）
├── workspace*/   # 用户内容（可选共享，含 skills/ 和 .agents/skills/）
├── skills/       # 托管技能（全局共享）
├── tools/        # 技能工具运行时（全局共享）
├── memory/       # 检索索引（派生数据，独立管理）
├── credentials/  # 认证凭据（全局）
└── ...
```

这种布局的优势：
- **备份/清理便捷**：要重建所有 memory 索引，删除 `memory/` 目录即可
- **安全边界清晰**：敏感凭据（`agents/*/agent/`）与用户内容（`workspace`）分离
- **灵活共享**：workspace 可共享而不影响 agent 身份隔离
- **生命周期独立**：索引可随时重建，凭据和会话需要持久保存

### 3.3 Gateway 启动时的 Agent 初始化

Gateway 启动时会遍历所有配置的 agent，为每个 agent 独立初始化 memory 后端（见 `src/gateway/server-startup-memory.ts`）：

```typescript
export async function startGatewayMemoryBackend(params) {
  const agentIds = listAgentIds(params.cfg);
  for (const agentId of agentIds) {
    // 每个 agent 独立检查 memorySearch 配置
    // 独立初始化 memory manager
    const { manager, error } = await getMemorySearchManager({
      cfg: params.cfg, agentId
    });
  }
}
```

## 附录：Q&A

### Q1: `agents/<agentId>/` 和 `workspace-<agentId>` 为什么要分成两个目录？

这两个目录的职责完全不同，是有意的**关注点分离**：

| | `agents/<agentId>/` | `workspace(-<agentId>)` |
|--|--|--|
| **定位** | 运行时内部状态（机器用的） | 用户可见的工作区（人和 agent 共用） |
| **典型内容** | `agent/auth-profiles.json`（认证凭据）、`agent/models.json`（模型配置）、`sessions/*.jsonl`（会话记录）、`qmd/`（QMD 索引状态） | `AGENTS.md`、`SOUL.md`（引导文件）、`memory/`（用户编写的记忆 Markdown）、用户项目文件 |
| **性质** | 不应手动编辑，是系统内部数据 | 用户可直接编辑，是工具的"工作目录" |

代码中有明确约束——**禁止两个 agent 共享同一个 agentDir**，否则会导致认证和 session 冲突（见 `src/config/agent-dirs.ts` `formatDuplicateAgentDirError`）。而 workspace 则相反——**允许**多个 agent 显式配置指向同一个 workspace（用于共享引导文件和记忆内容）。

简单说：`agents/<id>/` 是系统状态，`workspace` 是用户内容，生命周期和访问权限都不同，所以分开管理。

### Q2: 为什么 `memory/<agentId>.sqlite` 不放在 `agents/<agentId>/memory/` 下？

三个原因：

**a) 概念上，它是"派生索引"而非"源数据"**

`~/.openclaw/memory/<agentId>.sqlite` 存的是向量检索索引（embedding + chunking），是从 workspace 下的 `MEMORY.md` 和 `memory/*.md` 这些 Markdown 文件**构建出来的派生数据**。真正的"记忆"是那些用户编写的 Markdown 文件，SQLite 只是加速检索的缓存/索引。

**b) 与 agentDir 的管理约束不同**

`agents/<agentId>/agent/` 有严格的唯一性约束（不能共享），且里面存的是敏感的认证凭据。memory 索引没有这种安全顾虑，而且配置上需要更灵活——可以通过 `store.path` 配置项和 `{agentId}` 占位符自由指定路径（见 `src/agents/memory-search.ts` `resolveStorePath`）。

**c) 统一的顶层分类布局**

`~/.openclaw/` 下采用的是**按功能分顶层目录**的布局，让每类数据的备份、清理、迁移都更方便——比如要重建所有 memory 索引，直接删除 `memory/` 目录即可，不需要逐个去 `agents/*/` 下面找。

**设计总结**：目录设计遵循"按职责分离"的原则，而非"按 agent 聚合一切"。`agents/<id>/` 管身份和会话，`workspace` 管用户内容，`memory/` 管检索索引——各自独立管理，互不耦合。

### Q3: Agent 的技能（Skills）存在哪里？如何加载？

OpenClaw 的技能加载采用**多来源合并、按优先级覆盖**的架构。运行时通过 `loadSkillEntries()`（`src/agents/skills/workspace.ts`）从 6 个目录加载技能，同名技能按优先级高者覆盖低者。

#### 6 层技能来源（优先级从低到高）

| 优先级 | 来源标识 | 路径 | 说明 |
|--------|---------|------|------|
| 1（最低）| `openclaw-extra` | `openclaw.json` 中 `skills.load.extraDirs` 配置 + 插件技能目录 | 用户自定义额外目录和插件（`openclaw.plugin.json` 中声明 `skills` 字段）提供的技能 |
| 2 | `openclaw-bundled` | `<安装路径>/skills/`（仓库顶层的 `skills/`） | 随 OpenClaw 发行版打包的 50+ 内置技能（1Password、GitHub、Slack、天气等） |
| 3 | `openclaw-managed` | **`~/.openclaw/skills/`** | 运行时状态目录中的托管技能，通过 Gateway RPC 安装/管理 |
| 4 | `agents-skills-personal` | **`~/.agents/skills/`** | 用户主目录下的个人技能，跨所有项目共享 |
| 5 | `agents-skills-project` | **`<workspace>/.agents/skills/`** | 项目级技能，与工作区绑定 |
| 6（最高）| `openclaw-workspace` | **`<workspace>/skills/`** | 工作区技能，优先级最高，覆盖所有其它同名技能 |

对应源码：

```typescript
// src/agents/skills/workspace.ts — loadSkillEntries()

// Precedence: extra < bundled < managed < agents-skills-personal < agents-skills-project < workspace

for (const skill of bundledSkills)        { merged.set(skill.name, skill); }
for (const skill of managedSkills)        { merged.set(skill.name, skill); }
for (const skill of personalAgentsSkills) { merged.set(skill.name, skill); }
for (const skill of projectAgentsSkills)  { merged.set(skill.name, skill); }
for (const skill of workspaceSkills)      { merged.set(skill.name, skill); }
```

#### 技能文件结构

每个技能是一个目录，核心文件是 `SKILL.md`（Markdown 格式，可含 YAML frontmatter 元数据）：

```
<skillName>/
└── SKILL.md           # 技能定义（必须存在）
```

`SKILL.md` 的 frontmatter 可声明以下元数据（见 `src/agents/skills/types.ts`）：

- `os`：适用操作系统
- `requires.bins` / `requires.anyBins`：依赖的可执行文件
- `requires.env`：依赖的环境变量
- `requires.config`：依赖的配置项
- `primary-env`：主要环境变量（用于 `apiKey` 自动注入）
- `always`：始终加载（不受条件过滤）
- `command-dispatch`：命令调度模式（如 `tool`）
- `user-invocable` / `disable-model-invocation`：用户/模型调用策略

#### 技能过滤机制

加载后的技能还会经过多层过滤（见 `src/agents/skills/config.ts`）：

1. **全局开关**：`openclaw.json` 中 `skills.entries.<skillKey>.enabled: false` 可禁用单个技能
2. **内置白名单**：`skills.allowBundled` 可限制仅加载指定的内置技能
3. **运行时适配性**：根据操作系统、可执行文件、环境变量、配置项自动判断技能是否可用
4. **Agent 级白名单**：`agents.list[].skills?: string[]`（空数组 = 禁用全部技能）
5. **Prompt 容量限制**：`skills.limits.maxSkillsInPrompt`（默认 150）和 `skills.limits.maxSkillsPromptChars`（默认 30,000）控制注入系统提示的技能数量和字符总量；超出时自动降级为精简格式（仅名称 + 路径，省略描述）

#### 技能热重载

Gateway 运行期间，`ensureSkillsWatcher()`（`src/agents/skills/refresh.ts`）会通过 chokidar 监控所有技能目录中的 `SKILL.md` 文件变化，自动触发技能快照刷新，无需重启 Gateway。可通过 `skills.load.watch: false` 关闭。
